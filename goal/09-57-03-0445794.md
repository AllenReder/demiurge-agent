# 09-57-03 Bootstrap Slot Runtime Plan

## Hermes vs Demiurge comparison

- Hermes keeps prompt/context assembly in dedicated modules such as `agent/prompt_builder.py`, `agent/system_prompt.py`, and `agent/context_engine.py`.
- Hermes gateway stream events in `gateway/stream_events.py` explicitly describe transport only and say history/context ownership stays outside platform delivery.
- The useful lesson is not to copy Hermes' prompt cache or memory file layout. The useful seam is: context-producing lifecycle belongs in an agent/runtime module, while gateway/adapters only deliver presentation facts.
- Demiurge has a stronger Agent Slot concept than Hermes. However, after input/output slot context and slot effects were made runtime-owned, bootstrap slots still remain runner-local special handling:
  - `ModuleBootstrapClient` lives inside `runner.py`;
  - `SessionTurnStepRunner._ensure_bootstrap_context(...)` manually builds `BootstrapContext`;
  - runner invokes bootstrap slots, applies failure policy, emits bootstrap events, and writes the session bootstrap snapshot;
  - `run_turn(...)` has to know the full bootstrap phase algorithm before beginning the turn lifecycle.

## Breaking redesign target

Create a first-class bootstrap slot runtime:

- `BootstrapSlotRuntime` owns bootstrap slot context construction, slot invocation, failure policy, session snapshot persistence, and bootstrap events.
- `SessionTurnStepRunner` only asks the runtime to ensure the bootstrap snapshot exists.
- `SlotRuntime` remains the implementation that invokes authored Python slots.
- `SessionRuntime` remains the persistence module for the bootstrap snapshot.
- Agent Core, Agent Slot, and Package concepts are unchanged. `agent/bootstrap/*` package-installed slots still run as before.

## Deep module check

- Interface size: runner should call one method, `ensure(...)`, with core, capability, session id, core revision, workspace, and interaction metadata.
- Deletion test: deleting the new module would scatter bootstrap context construction, fragment collection, event naming, soft/hard failure policy, and snapshot writes back into runner.
- Hidden behavior: the module is not pass-through. It decides whether bootstrap is enabled, whether a snapshot already exists, how slot return values are ignored, how failures are handled, and when the session snapshot is committed.
- Adapter discipline: platform adapters and channels do not learn bootstrap semantics. Authored Agent Core and Packages still see only `ctx.bootstrap.add(...)`.

## Old interfaces and runner-local behavior to delete

- Move `ModuleBootstrapClient` out of `runner.py`.
- Delete runner `_ensure_bootstrap_context(...)`.
- Stop constructing `BootstrapContext` in runner.
- Stop emitting `bootstrap.*` events from runner.
- Stop writing bootstrap snapshots from runner.
- Keep `run_turn(..., use_bootstrap=True)` as the public runner behavior for now; it delegates to the new runtime.

## New seam placement

- Bootstrap slot seam: `demiurge/runtime/bootstrap.py`.
- Slot invocation seam: `demiurge/runtime/slots.py` remains the invocation adapter for authored slot Python code.
- Session persistence seam: `demiurge/runtime/session.py` remains the storage adapter for bootstrap snapshots.
- Context assembly seam: `demiurge/runtime/context.py` continues to read the stored bootstrap snapshot when building model messages.
- Runner seam: `demiurge/runtime/runner.py` composes runtime modules but stops owning bootstrap phase internals.

## Why this is framework redesign, not ordinary cleanup

- Bootstrap is a first-class Agent Slot phase, but the old shape made it the last phase still implemented directly by the runner.
- The new shape makes Agent Slot phases more coherent: bootstrap creates session context, input creates model context, output creates delivery/history effects, and each phase has a runtime owner.
- This reduces the runner from a lifecycle monolith toward a harness that composes deep runtime modules.
- Hermes is used only as design reference: context production is agent/runtime ownership, presentation stays in adapters, and prompt/context internals are not scattered through gateway code.

## New module shape

- Add `demiurge/runtime/bootstrap.py`:
  - `ModuleBootstrapClient`;
  - `BootstrapSlotRequest`;
  - `BootstrapSlotHost`;
  - `RunnerBootstrapSlotHost`;
  - `BootstrapSlotRuntime.ensure(...)`.
- Wire `SessionTurnStepRunner.__init__` to instantiate `self.bootstrap_slots`.
- Change `run_turn(...)` to call `self.bootstrap_slots.ensure(...)` when `use_bootstrap` is true.
- Keep authored SDK behavior unchanged: bootstrap slots still receive `BootstrapContext` with `ctx.bootstrap.add(...)`.

## Implementation steps

1. Add focused tests for `BootstrapSlotRuntime`:
   - disabled bootstrap does nothing;
   - existing snapshot is reused;
   - soft slot failure writes an empty snapshot and emits failure/completed events;
   - hard slot failure raises and does not write a snapshot;
   - slot return values are ignored while fragments are persisted.
2. Add `demiurge/runtime/bootstrap.py` and move `ModuleBootstrapClient` there.
3. Refactor `SessionTurnStepRunner` to instantiate `BootstrapSlotRuntime` and delete runner-local bootstrap logic.
4. Update imports and any tests that imported bootstrap helper objects from runner.
5. Keep package-installed bootstrap behavior unchanged.

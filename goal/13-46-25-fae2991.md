# Delivery Outbox Turn Ownership Plan

## Hermes vs Demiurge Comparison

Hermes routes foreground conversation output by session/thread metadata and
keeps background process/delegation completion ownership in watcher paths. A
platform message can belong to a thread without becoming a background process
task; process watchers separately own background status and completion
notifications.

Demiurge now keeps foreground turns out of the task ledger and narrowed the
control plane to real task specs. The next vocabulary leak is delivery/outbox:
delivery intents still store `task_id=turn_id`, and transient delivery durable
work uses `parent_work_id=turn_id`. That makes a foreground turn look like a
task/work parent again, just through the delivery path instead of the task
control plane.

## Framework Shape Decision

Delivery rows should express turn ownership directly:

- `owner_turn_id` is the delivery/outbox correlation field for foreground turn
  output.
- `delivery.send` durable work keeps `owner_turn_id` but does not set
  `parent_work_id` to a foreground turn id.
- `append_delivery_message()` accepts `turn_id` once and derives delivery
  ownership from it.
- Queries that want deliveries for a turn use `owner_turn_id`, not `task_id`.

This makes the seam explicit: delivery belongs to a session turn, while task
ledger rows belong to detached controllable work.

## Delete Or Merge

- Delete `task_id` from `SessionRuntime.append_delivery_message()`.
- Delete `task_id` from `delivery.queued` payloads for delivery intents.
- Delete `task_id` from `delivery.send` durable work payloads.
- Delete `parent_work_id=turn_id` for foreground turn delivery work.
- Replace the outbox projection column `task_id` with `owner_turn_id`.
- Update module delivery, tests, and docs that query delivery rows by `task_id`.

## Behavior To Keep

- Delivery still records assistant history before queuing persisted deliveries.
- Transient delivery still queues outbox rows without assistant history.
- `DeliveryRuntime` still claims `delivery.send` durable work and updates outbox
  status through `queued -> sending -> sent/failed/unknown/unrouted`.
- Delivery rows remain queryable by `delivery_id` and now by `owner_turn_id`.
- Runtime task tables and task completion behavior stay untouched.

## Why This Is A Demiurge Framework Step

This removes another old assumption that foreground turn ids can be reused as
task/work ids. The outbox becomes a deeper delivery module because callers no
longer need to know a fake task correlation convention. Delivery ownership is
local to session/turn projection, while real task parentage remains local to
the task ledger.

## Implementation Steps

1. Change outbox projection schema and `delivery.queued` projection code from
   `task_id` to `owner_turn_id`.
2. Refactor `SessionRuntime.append_delivery_message()` and
   `ModuleDeliveryRuntime` delivery events/work specs to use `owner_turn_id`.
3. Update runtime tests and queries from `task_id` to `owner_turn_id`.
4. Update delivery/runtime developer docs where the outbox ownership field is
   described or implied.
5. Validate delivery, session, module-delivery, slot-effect, and outbox tests;
   compile touched files; grep stale delivery `task_id` ownership; run scoped
   diff checks.
6. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

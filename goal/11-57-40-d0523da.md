# 11-57-40 Operator Gateway Lifecycle Plan

## Global architecture review

The previous round moved channel conversation lifecycle into
`ConversationLifecycleRuntime`. The next framework-shape problem is the local
operator gateway:

- `demiurge/ui_gateway/bridge.py` still owns active-task lifecycle,
  busy-input queue/interrupt behavior, finish/drain-next, task-worker
  subscription, background completion enqueue, and pending completion merge.
- The same lifecycle rules now exist behind the channel runtime seam, so TUI is
  the remaining adapter that still composes host lifecycle primitives directly.
- TUI must remain an operator-console adapter, not a messaging channel. It owns
  status events, prompt/approval modals, history snapshots, local commands, and
  UI payload shape.

## Hermes vs Demiurge comparison

- Hermes gateway keeps running-session slots, busy input handling, pending
  follow-up drain, and task/completion continuation in gateway lifecycle code.
- Hermes platform adapters and operator-facing gateways keep their
  presentation/transport concerns separate from those lifecycle decisions.
- Current Demiurge already has a channel-side lifecycle runtime. Leaving TUI on
  the old hand-composed path would create two host lifecycle dialects and keep
  future task/session changes coupled to UI code.

## Breaking redesign target

Extend `ConversationLifecycleRuntime` so it can serve operator adapters as well
as messaging-channel adapters:

- Keep host-owned behavior in the lifecycle runtime:
  - active task start/finish/drain-next;
  - busy queue/interrupt decisions;
  - pending background completion merge;
  - task-worker completion subscription and enqueue.
- Add an optional `after_completion_enqueue` hook so operator UIs can refresh
  status after a background completion is queued or started.
- Add a small turn-finished hook and drain predicate so operator UIs can refresh
  status after lifecycle-owned finish/drain and can prevent queued input from
  draining during shutdown.
- Migrate `TuiInteractionBridge` to the lifecycle runtime while preserving
  TUI-specific behavior:
  - user/status/history/command events;
  - prompt and approval flows;
  - interrupt notices;
  - scheduler startup/shutdown;
  - local session resume/new behavior.

## Deep module check

- Interface size: TUI should call `remember_route(...)`, `merge_pending(...)`,
  `handle_busy(...)`, `accept_inbound(...)`, `run_state_turn(...)`, and
  `cancel_active(...)`; it should not manipulate finish/drain/subscription or
  completion claim/enqueue itself.
- Deletion test: deleting the lifecycle module would re-scatter queue,
  active-task, completion-claim, task-worker subscription, and drain-next logic
  across Text, Telegram, and TUI adapters.
- Hidden behavior: the module owns real lifecycle sequencing. TUI hooks only
  decorate lifecycle transitions with operator-console events.
- Adapter discipline: TUI remains an operator-console adapter. It does not
  become a channel, and channel-specific concepts like allowlists or native
  Telegram media do not move into the lifecycle runtime.

## Old interfaces and adapter-local behavior to delete

- Delete direct `CompletionDeliveryRuntime` construction from
  `TuiInteractionBridge`.
- Delete direct task-worker subscription ownership from `TuiInteractionBridge`.
- Delete TUI-local `_on_task_completion(...)`,
  `_enqueue_task_completion(...)`, and `_drain_next_queued_input(...)`.
- Remove finish/drain logic from `_run_inbound(...)`.
- Replace direct `ConversationTurnController` usage in submit/interrupt/running
  with lifecycle runtime calls where the bridge is not implementing a local
  command detail.

## New seam placement

- `demiurge/runtime/conversation_lifecycle.py`
  - add optional `after_completion_enqueue` hook;
  - add optional `after_turn` and `should_drain` hooks for operator status and
    shutdown behavior;
  - keep callback surface small and adapter-neutral.
- `demiurge/ui_gateway/bridge.py`
  - construct one lifecycle runtime with `channel="tui"`,
    `merge_owner_id="bridge:tui:stored"`,
    `enqueue_owner_id="bridge:tui:enqueue"`, and `fallback_source="local"`;
  - keep a single operator ingress state;
  - use lifecycle runtime for start, busy, cancel, completion merge, and
    background completion enqueue;
  - close lifecycle subscriptions during shutdown.

## Why this is framework redesign, not ordinary cleanup

- It collapses the remaining operator/channel split in host lifecycle semantics
  without collapsing their adapter concepts.
- It gives future task/session/schedule changes one lifecycle module to change
  instead of three adapter-local implementations.
- It moves Demiurge closer to a Hermes-style gateway runtime: adapters present
  interaction surfaces; runtime owns conversation lifecycle.

## Implementation steps

1. Extend `ConversationLifecycleRuntime` with an
   `after_completion_enqueue` callback and tests.
2. Migrate `TuiInteractionBridge` to construct and use the lifecycle runtime.
3. Remove TUI-local completion subscription/enqueue and finish/drain code.
4. Preserve existing TUI command, prompt, approval, status, and history behavior.
5. Run focused runtime and UI gateway tests.

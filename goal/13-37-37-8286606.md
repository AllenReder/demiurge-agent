# Task Control Plane Interface Narrowing Plan

## Hermes vs Demiurge Comparison

Hermes keeps foreground conversation/session flow separate from background
watchers and scheduled/process work. The gateway drains watch events, process
completions, and async delegation completions through specific watcher paths
rather than pushing every runtime action through a single "action becomes task"
interface. That makes route/session metadata responsible for conversation
delivery, while process/delegation watchers own detached work status.

Demiurge just removed foreground turns from the task ledger, but
`RuntimeControlPlane` still exposes a shallow interface named `ActionSpec` /
`ActionKind` / `submit()`. The implementation always appends
`task.submitted`, returns a `TaskHandle`, and projects into the `tasks` table.
The broad kind list still claims tool calls, MCP calls, delivery sends,
approval requests, state patches, and artifact writes are valid task submissions,
even though those are not controllable detached tasks in the current runtime.

## Framework Shape Decision

Make the external control-plane seam task-specific:

- `TaskSpec` describes a task-ledger row, not any runtime action.
- `TaskSource` describes the host or model actor that created or updated a
  task.
- `RuntimeControlPlane.submit_task()` is the only task submission entry point.
- Valid task ledger kinds are `agent.spawn`, `terminal.exec`, `evolver.run`,
  and `schedule.fire`.
- Non-task runtime facts should use their own projection/event modules or
  `record_events()`, not a fake task submission.

This keeps `RuntimeControlPlane` as a deep module for detached work instead of a
generic pass-through event wrapper. Deleting it would now push task lifecycle,
logs, completion events, query/read/control, and schedule-fire status back into
task producers.

## Delete Or Merge

- Delete `ActionKind`.
- Delete `ActionSpec`.
- Delete `ActionSource`.
- Delete unused `ActionResult`.
- Delete `RuntimeControlPlane.submit()`.
- Replace with `TaskKind`, `TaskSpec`, `TaskSource`, and
  `RuntimeControlPlane.submit_task()`.
- Update active callers in `RuntimeTaskWorker` and scheduler code.
- Update runtime tests and developer docs.

## Behavior To Keep

- `RuntimeTaskWorker.start_task()` still submits `agent.spawn`, `terminal.exec`,
  and `evolver.run` task rows and emits completion notifications.
- Scheduler fire claims still appear as `schedule.fire` task rows and can be
  marked started/succeeded/failed by the scheduler.
- Task logs, task reads, task query filters, task stream, completion-ready
  events, and completion acknowledgements keep their existing projection shape.
- `RuntimeTaskWorker` remains stricter than the control plane: it executes only
  active background task kinds and continues to reject `schedule.fire` as a live
  worker kind.

## Why This Is A Demiurge Framework Step

This removes the leftover host vocabulary that implied "any action can become a
task". The task ledger becomes a narrower, more honest module with a smaller
interface and stronger locality: producers that create task rows must be
creating detached, inspectable host work. Delivery, approval, tool-call, state,
and artifact behavior can later deepen through their own modules instead of
inheriting task semantics by accident.

## Implementation Steps

1. Refactor `demiurge/runtime/control.py` from Action names to Task names and
   add runtime validation for task-ledger kinds.
2. Update `RuntimeTaskWorker` and scheduler callers to use `TaskSpec`,
   `TaskSource`, and `submit_task()`.
3. Update tests to assert `TaskSpec` rejects non-task kinds and to stop using
   control-plane task submission for non-task rows.
4. Update developer docs and zh-CN website mirror to describe `TaskSpec -> Task`
   and the valid task-ledger kinds.
5. Validate focused runtime tests, compile touched files, grep stale Action names,
   and run scoped diff checks.
6. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

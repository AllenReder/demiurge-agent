## Reference Comparison

### Hermes reference

Hermes documents session lifecycle as gateway-owned state: session identity, transcript persistence, reset/resume behavior, and restart recovery live below platform-specific adapters. The important lesson is that transcript recovery is a runtime/session read-model, not presentation glue embedded in a single UI adapter.

The transferable design is not Hermes's storage schema. Demiurge should keep its SQLite/session-event model, but the rule that turns persisted session messages and runtime events into visible history items should live behind a runtime module interface.

### Demiurge current state

This branch now has runtime read-model modules for slash commands, sessions, runtime status, tool display, shared text formatting, and trace/event commands.

The remaining shallow history cluster is in `demiurge/ui_gateway/bridge.py`:

- `_emit_history_snapshot(...)` emits `interaction.history`;
- `_history_items(...)` reads session messages and session event logs;
- `_history_items(...)` maps visible `user` / `assistant` / `system` messages into transcript items;
- `_history_items(...)` suppresses tool history in quiet mode and reconstructs tool cards in summary/full modes;
- `_tool_history_events(...)` rebuilds tool-call metadata from `actions.requested` and `action.result` events.

Deletion test: deleting a shared history snapshot module today would force the same message metadata, tool event projection, and tool display-mode rules back into TUI or any future dashboard/Web UI.

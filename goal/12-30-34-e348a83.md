# 12-30-34 Agent Slot Runtime Collapse Plan

## Global architecture review

The channel and operator gateway lifecycle work is now concentrated behind
`ConversationLifecycleRuntime`. A wider review shows the next framework-shape
problem is the Agent Slot runtime seam:

- `demiurge/runtime/slots.py` owns phase pipeline scheduling.
- `demiurge/runtime/slot_execution.py` owns single-slot execution, skill
  activation, slot-end delivery scheduling, soft/hard failure policy, ignored
  return handling, and output history refresh.
- `RunnerSlotPipelineHost` and runner methods
  `run_input_pipeline_slot(...)`, `run_output_pipeline_slot(...)`, and
  `flush_slot_background_items(...)` only forward calls between those two
  modules.

This makes the Agent Slot runtime shallow and spread across runner glue. The
module interface should be "run this input/output Agent Slot phase", not "ask
runner to call a different module for every slot".

## Hermes vs Demiurge comparison

- Hermes keeps the agent conversation loop responsible for prompt/context/tool
  orchestration while helper modules provide concrete capabilities behind
  smaller interfaces.
- Demiurge should keep Agent Slots as an authored surface, but the host runtime
  should own slot scheduling, execution, effect application, skill activation,
  and background flushing as one coherent module.
- Current Demiurge's split makes runner know too much about internal slot
  execution sequencing, even though runner is not the adapter that varies.

## Breaking redesign target

Collapse `SlotExecutionRuntime` into `SlotPipelineRuntime`:

- `SlotPipelineRuntime` owns serial and parallel phase scheduling.
- It also owns per-slot context construction, invocation, skill activation,
  effect scheduling/failure, ignored return events, and output history refresh.
- Delete `demiurge/runtime/slot_execution.py`.
- Delete `RunnerSlotPipelineHost`.
- Delete runner wrapper methods for single-slot execution and background item
  flushing.

Agent Core, Agent Slot, and Package concepts stay unchanged. The authored
`agent/bootstrap`, `agent/input`, `agent/output`, slot metadata, ctx shape, and
capability enforcement remain host-controlled and author-visible as before.

## Deep module check

- Interface size: runner should construct `SlotPipelineRuntime` with its stable
  dependencies and call only `run_input(...)` / `run_output(...)`.
- Deletion test: deleting the merged slot runtime would scatter scheduling,
  context construction, invocation, effect handling, skill activation, and
  failure policy back across runner and multiple modules. The module earns its
  place.
- Hidden behavior: the runtime hides real behavior, not pass-through glue.
- Adapter discipline: runner retains host dependency wiring; authored Agent Core
  owns files; packages still provide slot/tool/skill assets. Slot runtime owns
  execution semantics.

## Old interfaces and files to delete

- Delete `demiurge/runtime/slot_execution.py`.
- Delete `RunnerSlotPipelineHost` and `SlotPipelineHost` from
  `demiurge/runtime/slots.py`.
- Delete `SessionTurnStepRunner.run_input_pipeline_slot(...)`,
  `run_output_pipeline_slot(...)`, and `flush_slot_background_items(...)`.
- Replace `tests/runtime/test_slot_execution.py` with coverage in
  `tests/runtime/test_slot_pipeline.py`.

## New seam placement

- `demiurge/runtime/slots.py`
  - owns `SlotPipelineRuntime`;
  - takes `slot_runtime`, `slot_context`, `slot_effects`, event emission,
    background task tracking, and history refresh as constructor dependencies;
  - keeps `SlotRuntime` as the low-level authored Python callable invoker.
- `demiurge/runtime/runner.py`
  - wires slot runtime dependencies only;
  - calls phase-level `run_input` / `run_output`.

## Why this is framework redesign, not ordinary cleanup

- It removes a runner-to-runtime-to-runner loop and moves Agent Slot lifecycle
  semantics behind one deep module.
- It is a concrete step toward a new Demiurge framework shape where runner is
  composition/root wiring and runtime modules own host lifecycle.
- It makes the Agent Slot contract easier to reason about: tests cross the same
  phase-level interface that runner uses.

## Implementation steps

1. Move single-slot execution methods from `slot_execution.py` into
   `SlotPipelineRuntime`.
2. Replace host callbacks in `SlotPipelineRuntime` with direct constructor
   dependencies.
3. Update `SessionTurnStepRunner` initialization order and remove wrapper
   methods/imports.
4. Merge slot execution tests into `tests/runtime/test_slot_pipeline.py`.
5. Delete `slot_execution.py` and its dedicated test file.

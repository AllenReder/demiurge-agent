## Reference Comparison

### Hermes reference

Hermes keeps gateway lifecycle rules behind named Modules:

- `gateway/stream_dispatch.py` routes typed events through one dispatcher, while platform adapters only render platform-specific output.
- `tui_gateway/event_publisher.py` isolates best-effort queueing and drop behavior behind a transport Module.
- `gateway/session.py` packages platform source and session routing facts into explicit records.

The transferable lesson is that once a lifecycle rule is shared by multiple adapters, the rule should live behind one Module Interface. Demiurge should not copy Hermes implementation code, but it should keep adapter-specific presentation out of shared turn intake policy.

### Demiurge current state

This branch now has:

- `TurnEngine` for model/tool loop orchestration.
- `SlotPipelineRuntime` for input/output phase scheduling.
- `SlotContextRuntime` for authored `ctx.*` construction.
- `TurnIO` for model-step transcript and interaction-item assembly.
- `ModuleDeliveryRuntime` for authored delivery request application.
- `ChildAgentRuntime` for child-agent lifecycle.
- `CompletionInbox` for background completion claim, inbound construction, merge metadata, and ack.

The remaining shallow cluster is inbound queue policy:

- `demiurge/ui_gateway/bridge.py` owns `_queued_inputs`, next-input selection, user-first prioritization, and queued completion merge.
- `demiurge/channels/base.py` owns the same queue drain, completion-preserving clear, user-first prioritization, and completion merge for generic text channels.
- `demiurge/channels/telegram/bridge.py` repeats the same implementation for Telegram.

Deletion test: deleting any one adapter's queue helper would force the same user-vs-completion selection and completion-preserving clear logic to reappear locally. Three real adapters justify the seam.

## This Loop Scope

Extract an `InboundQueueRuntime` Module that owns queue storage, queued inbound selection, queued completion merge, and completion-preserving clear.

This is behavior-preserving. It changes internal Module placement and adapter call sites only; it does not change Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, task tool semantics, completion event schema, busy mode meanings, prompt behavior, approval behavior, command behavior, TUI rendering, or Telegram delivery behavior.

## Planned Code Changes

- Add `demiurge/runtime/ingress.py`.
- Add `InboundQueueRuntime` with a small Interface:
  - `put(...)`;
  - `put_nowait(...)`;
  - `empty()`;
  - `qsize()`;
  - `next_inbound()`;
  - `clear(preserve_completions=...)`;
  - `merge_completions_into(inbound, stored_completions=...)`.
- Move queue rules into that Module:
  - drain all pending queue entries for selection;
  - prefer the first non-completion user inbound;
  - if a user inbound is selected, merge queued background completions into it;
  - preserve remaining non-selected inbounds in queue order;
  - clear ordinary inputs while optionally preserving background completions;
  - merge stored completions plus queued completions into a new user inbound while leaving ordinary queued inputs in place.
- Update `TuiInteractionBridge`:
  - replace `asyncio.Queue[InteractionInbound]` with `InboundQueueRuntime`;
  - delegate `_next_queued_input(...)` and `_merge_pending_completions_into(...)` to the queue Module;
  - remove direct queue-drain/merge logic.
- Update `TextConversationState` and `TextChannelBridgeBase`:
  - replace raw `asyncio.Queue` with `InboundQueueRuntime`;
  - delegate `_clear_queue(...)` and `_next_queued_input(...)`;
  - keep busy-mode send/interrupt behavior adapter-owned.
- Update `TelegramConversationState` and `TelegramInteractionBridge` the same way.
- Add focused tests in `tests/runtime/test_ingress.py`:
  - next inbound prefers user input over queued completions and merges completions into it;
  - completion-only queue returns the completion unchanged;
  - `clear(preserve_completions=True)` removes ordinary inputs and preserves completions;
  - `merge_completions_into(...)` merges stored plus queued completions while preserving ordinary queued inputs.

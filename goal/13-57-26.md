# Artifact Turn Ownership Plan

## Hermes vs Demiurge Comparison

Hermes treats files, media, screenshots, and local delivery outputs as artifacts
of the active session/tool flow. They can be referenced by path or metadata, but
foreground conversation turns do not need to be promoted into background task
ids just to own those files. Background jobs and process watchers own their own
status separately.

Demiurge has already split foreground turns from the task ledger and made
delivery outbox rows owned by `owner_turn_id`. The next framework-shape leak is
artifact storage: `ModuleDeliveryRuntime` emits `artifact.stored` with
`task_id=turn_id`, and `RuntimeStore` projects artifacts into a table keyed by a
`task_id` column. That preserves the old "turn ids can masquerade as task ids"
convention inside the artifact projection.

## Framework Shape Decision

Artifacts created by foreground output delivery should be turn-owned:

- `artifact.stored` carries `owner_turn_id` for foreground turn ownership.
- The `artifacts` projection stores `owner_turn_id`, not `task_id`.
- Artifact metadata can still contain `session_id`, `turn_id`, media metadata,
  summaries, and authored metadata.
- Real task-produced artifacts can later use an explicit task-owned projection
  or a separate task owner field; this loop only removes the foreground turn as
  fake task owner.

This keeps artifact ownership local to the session/turn delivery module, while
task ownership remains local to detached runtime tasks.

## Delete Or Merge

- Delete `task_id` from foreground `artifact.stored` payloads.
- Replace the `artifacts.task_id` projection column with `owner_turn_id`.
- Update artifact tests to assert `owner_turn_id`.
- Update delivery/artifact developer docs to describe turn-owned artifacts.

## Behavior To Keep

- Output modules can still request artifact delivery.
- Host still owns artifact paths, summaries, metadata, and persistence.
- Delivery metadata still includes artifact descriptors for channel rendering.
- Runtime event stream still records `artifact.stored` events.
- Existing task tables, task logs, and task completion behavior are untouched.

## Why This Is A Demiurge Framework Step

This removes the last direct foreground delivery path found in this pass that
uses task vocabulary to describe turn-local runtime data. The artifact projection
becomes a deeper module because callers learn the real ownership model:
artifacts belong to a turn unless a future explicit task-artifact seam says
otherwise.

## Implementation Steps

1. Change `ModuleDeliveryRuntime` to emit `owner_turn_id` in `artifact.stored`.
2. Change `RuntimeStore` artifact projection schema and projector to store
   `owner_turn_id`.
3. Update module delivery tests to assert turn-owned artifact rows.
4. Update delivery/reference docs and zh-CN mirrors where artifact ownership is
   described.
5. Validate module delivery, slot effects, outbound delivery, and store-adjacent
   runtime tests; compile touched files; grep stale foreground artifact
   `task_id` ownership; run scoped diff checks.
6. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

# 11-28-18 Slot Effect Runtime Plan

## Hermes vs Demiurge comparison

- Hermes keeps platform/gateway adapters focused on transport and route facts, while tool/output side effects are interpreted inside the agent runtime and gateway delivery layers.
- Current Demiurge already has `SlotEffectRuntime` for delivery requests and pending interaction items, but `SessionTurnStepRunner` still owns the legacy authored effect interpreter:
  - normalizes raw dict / `DeliverEffect` / `EffectRequest`;
  - rewrites `append_assistant_message` into delivery effects;
  - enforces effect tool capabilities;
  - executes `tool_call` and `evolve_core` effects;
  - emits `effect.ignored` and `capability.denied`.
- That means the Agent Slot authored contract is not fully behind the Slot Effect module. The runner still knows too much about effect types.

## Breaking redesign target

Move authored effect interpretation into `SlotEffectRuntime`:

- `SlotEffectRuntime.handle_effects(...)` owns legacy effect normalization and dispatch.
- `SlotEffectRuntime.normalize_effect(...)` owns conversion from old dict/`DeliverEffect` values into `EffectRequest`.
- `SlotEffectRuntime.normalize_context_items(...)` owns legacy context contribution normalization if callers still need it.
- Runner only supplies a small host adapter callback for executing a tool effect, because actual tool execution remains host-owned and capability-controlled.
- Runner no longer contains effect type condition ladders.

Agent Core, Agent Slot, and Package concepts remain unchanged. Slot authors still use the same conceptual capabilities: delivery, context contribution, tool call effects, and capability enforcement through host `ctx`.

## Deep module check

- Interface size: runner should call `slot_effects.handle_effects(...)` or expose no effect internals at all.
- Deletion test: deleting the module would scatter delivery conversion, effect normalization, capability denied handling, ignored-effect events, and tool-effect delivery result shaping back across runner and slot execution callers.
- Hidden behavior: this module has real behavior. It decides how authored effect values become runtime deliveries and which effects are ignored or denied.
- Adapter discipline: the host tool callback is an adapter. The effect module owns policy; the runner only bridges to host-owned tool execution.

## Old runner-local behavior to delete or move

- Move `_handle_effects(...)` from runner to `SlotEffectRuntime`.
- Move `_normalize_effect(...)` from runner to `SlotEffectRuntime`.
- Move `_normalize_context_items(...)` from runner to `SlotEffectRuntime`.
- Remove `DeliverEffect` / `EffectRequest` / `CapabilityDenied` effect-specific imports from runner unless still required elsewhere.

## New seam placement

- Existing module: `demiurge/runtime/slot_effects.py`
  - add async `handle_effects(...)`;
  - add `normalize_effect(...)`;
  - add `normalize_context_items(...)`;
  - accept `execute_tool_effect` and `emit_event` callbacks.
- Runner composition:
  - construct `SlotEffectRuntime(..., execute_tool_effect=..., emit_event=...)`;
  - provide `_execute_slot_effect_tool(...)` as the host tool adapter.

## Why this is framework redesign, not ordinary cleanup

- This closes a mismatch created by prior slot-runtime extraction: the execution module runs slots, but the effect contract still leaked into the runner.
- Agent Slot becomes a clearer authored contract: slot code produces delivery/tool/context effects; the Slot Effect module interprets them; host tool execution remains behind a narrow adapter.
- This moves another chunk of harness policy out of the runner facade and toward a framework shape where Slot runtime modules can evolve independently.

## Implementation steps

1. Extend `SlotEffectRuntime` with effect normalization and async `handle_effects(...)`.
2. Add a runner tool-effect adapter callback.
3. Remove runner-local effect normalization/handling internals or replace with a narrow delegate only if needed.
4. Update tests to target `SlotEffectRuntime` directly instead of runner private methods.
5. Run focused slot effect/modular IO/slot execution tests.

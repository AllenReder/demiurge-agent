## Reference Comparison

### Hermes reference

Hermes documents active-session queueing, overflow promotion, and restart recovery as gateway/session lifecycle rules below platform adapters. Its delivery module also centralizes delivery routing concerns instead of asking every platform adapter to reimplement output policy.

The transferable design is not Hermes's exact data structures. Demiurge should keep `RuntimeTaskWorker`, `CompletionInbox`, `ConversationIngressState`, and `InteractionInbound`, but completion intake should be part of the conversation turn module interface rather than copied into each adapter.

### Demiurge current state

This branch has already extracted runtime read-model modules for commands, status, tool display, text formatting, event commands, history display, and outbound delivery planning.

The remaining shallow background completion cluster is in:

- `demiurge/ui_gateway/bridge.py`;
- `demiurge/channels/base.py`;
- `demiurge/channels/telegram/bridge.py`;
- `demiurge/runtime/ingress.py`.

The three adapters still repeat the same rules:

- claim stored pending completion events for the current session and merge them into a user inbound;
- claim a single completion event after a background task finishes;
- enqueue the synthetic completion inbound if a turn is running, drain it if user input is queued, or start it when idle;
- keep adapter-specific notification/platform behavior outside the rule.

Deletion test: deleting a shared completion-intake interface today would force claim/merge/enqueue rules back into TUI, generic text channels, and Telegram.

## This Loop Scope

Deepen `ConversationTurnController` so it owns background completion intake rules.

This is behaviour-preserving. It does not change task completion event schema, `InteractionInbound` metadata shape, queue ordering, user-visible completion text, Telegram delivery behavior, TUI events, Agent Core layout, Agent Slot manifests, Package recipes, or authored SDK methods.

- Add `ConversationTurnController.enqueue_completion_event(...)`.
- Update `TuiInteractionBridge`, `TextChannelBridgeBase`, and `TelegramInteractionBridge` to call these runtime methods instead of hand-rolling claim/merge/enqueue.

  - route-required completion events are ignored without claiming durable state;

## Expected Benefits

- Completion intake becomes one deeper runtime module interface.
- Locality improves: claim/merge/enqueue bugs concentrate in `runtime/ingress.py`.
- Leverage improves: TUI, generic text channels, and Telegram share the same completion intake rule.
- Adapters keep adapter responsibilities: source lookup, platform notices, and dispatch.

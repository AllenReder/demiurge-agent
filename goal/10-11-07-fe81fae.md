# 10-11-07 Delegation Tool Runtime Plan

## Hermes vs Demiurge comparison

- Hermes treats background work and forked review as host-owned lifecycle. `agent/background_review.py` triggers controlled background review after a real final response, and Hermes gateway/background commands keep execution control outside platform rendering.
- Hermes' useful lesson is not its exact forked `AIAgent` or tool whitelist implementation. The useful seam is: model-facing work-control commands should map to a host runtime module, not stay embedded in the agent loop or platform adapter.
- Demiurge already has `ChildAgentRuntime` for child execution and `RuntimeTaskWorker` for durable tasks, but the model-facing tools are still a runner-local switch:
  - `DELEGATION_TOOL_NAMES` lives in `runner.py`;
  - `execute_tool(...)` special-cases delegation tools before `ToolRuntime`;
  - `handle_delegation_tool(...)`, `_task_status(...)`, `_task_control(...)`, `_yield_until(...)`, and `_task_view(...)` live on the runner;
  - task-control capability checks and task payload formatting are runner behavior.

## Breaking redesign target

Create a model-facing delegation tool seam:

- `DelegationToolRuntime` owns `delegate_task`, `task_status`, `task_control`, and `yield_until` handling.
- `ChildAgentRuntime` continues to own child-agent execution and validation for `delegate_task`.
- `RuntimeTaskWorker` continues to own task records, cancellation, waiting, logs, and kind validation.
- `SessionTurnStepRunner` stops exposing delegation/task tool helper methods and only delegates to a small runtime interface from `execute_tool(...)`.

Agent Core, Agent Slot, and Package concepts are unchanged. The authored tool names and schemas stay the same in this loop.

## Deep module check

- Interface size: runner should call `self.delegation_tools.can_handle(name)` and `self.delegation_tools.execute(call, core, turn, capability)`.
- Deletion test: deleting the module would scatter capability checks, missing-task errors, JSON payload formatting, task wait timeout behavior, cancellation handling, and child delegation dispatch back into runner.
- Hidden behavior: the module does real behavior. It gates model-facing task control, consumes completions through `RuntimeTaskWorker.wait(..., consume_completion=True)`, formats model-visible payloads, and delegates child spawning through `ChildAgentRuntime`.
- Adapter discipline: channels, TUI, and gateway do not own task tool semantics. They only render interaction/output events.

## Old interfaces and runner-local behavior to delete

- Delete `DELEGATION_TOOL_NAMES` from `runner.py`.
- Delete runner `handle_delegation_tool(...)`.
- Delete runner `_delegate_task(...)`.
- Delete runner `_delegation_context(...)`.
- Delete runner `_task_status(...)`.
- Delete runner `_task_control(...)`.
- Delete runner `_yield_until(...)`.
- Delete runner `_task_view(...)`.

## New seam placement

- Delegation tool seam: `demiurge/runtime/delegation_tools.py`.
- Child-agent execution seam: `demiurge/runtime/child_agents.py` remains the child run/spawn implementation.
- Durable task seam: `demiurge/runtime/tasks.py` remains durable task storage/control/wait/log implementation.
- Runner seam: `demiurge/runtime/runner.py` composes tool runtime and delegation tool runtime but no longer owns individual delegation tool commands.

## Why this is framework redesign, not ordinary cleanup

- Background child-agent work is a first-class framework capability, but the model-facing work-control tools are still runner-specific branches.
- Moving them behind `DelegationToolRuntime` is a step toward a larger `AgentWorkRuntime` that can later absorb completion inbox, schedule-triggered work, background review, and session-scoped work control.
- Hermes is used only as design reference: background/forked work is host-owned lifecycle, not platform delivery and not a runner switch.

## New module shape

- Add `demiurge/runtime/delegation_tools.py`:
  - `DELEGATION_TOOL_NAMES`;
  - `DelegationToolHost`;
  - `RunnerDelegationToolHost`;
  - `DelegationToolRuntime.can_handle(...)`;
  - `DelegationToolRuntime.execute(...)`.
- Wire `SessionTurnStepRunner.__init__` to instantiate `self.delegation_tools`.
- Change `SessionTurnStepRunner.execute_tool(...)` to call the delegation runtime before ordinary tool execution.

## Implementation steps

1. Add focused tests for `DelegationToolRuntime`:
   - unknown delegation tool returns an unsupported-tool result;
   - `task_status` requires `task_id`;
   - unsupported `task_control` command is rejected;
   - missing background task produces the existing `background task not found` message;
   - capability denial is converted to a tool error.
2. Add `demiurge/runtime/delegation_tools.py`.
3. Move task-control helper logic from runner to `DelegationToolRuntime`.
4. Refactor runner to instantiate and call the new runtime, then delete old helper methods and constant.
5. Keep `delegate_task` behavior delegated to `ChildAgentRuntime.handle_delegate_task(...)`.

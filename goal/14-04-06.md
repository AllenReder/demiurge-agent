# Tool Call Turn Step Projection Plan

## Hermes vs Demiurge Comparison

Hermes treats ordinary tool lifecycle telemetry as part of the active
session/turn flow. Its observer and middleware contracts include session,
turn, tool-call, and sometimes task identity, but task identity is meaningful
only when the tool execution is happening inside a real delegated or managed
task context.

Demiurge has already removed foreground turns from the task ledger, but the
`tool_calls` projection still exposes a `task_id` column. Foreground model-step
tool calls currently emit only `turn_id`, so that column is usually empty. The
projection therefore preserves the old task-shaped interface without adding
leverage for callers.

## Framework Shape Decision

Foreground tool-call records should be turn/step-owned:

- `tool.call.started` and terminal tool-call events carry `turn_id` and
  `step_id`.
- The `tool_calls` projection stores `turn_id` and `step_id`, not `task_id`.
- Detached task tool activity can later use an explicit task-owned tool-call
  projection or metadata field if a real task observer seam needs it.
- The task ledger remains reserved for detached, observable, cancellable work.

This makes the tool-call projection a deeper module: callers can query the
actual foreground execution locality without learning an unused task-shaped
column.

## Delete Or Merge

- Delete `tool_calls.task_id`.
- Add `tool_calls.step_id`.
- Include `step_id` in tool-call runtime events emitted by `TurnEngine`.
- Update runtime projection tests to assert turn/step ownership.
- Update developer docs and zh-CN mirrors to describe tool-call records as
  session/turn/step projections rather than task facts.

## Behavior To Keep

- TUI and channel delivery still receive running and finished tool-call items.
- Tool result messages still persist with provider-compatible `tool_call_id`.
- Tool-call status, arguments, result payload, and timestamps still project into
  SQLite.
- Background task tools still return and control real `task_id` values.

## Why This Is A Demiurge Framework Step

This removes another shallow compatibility-shaped interface from the foreground
turn path. The seam between session execution and detached task execution
becomes sharper: session/tool projections own model-step tool activity, while
the task ledger owns only detached runtime work.

## Implementation Steps

1. Change `TurnEngine` to include `step_id` in `tool.call.started`,
   `tool.call.completed`, and `tool.call.failed` runtime events.
2. Change `RuntimeStore` to project `tool_calls.step_id` and drop
   `tool_calls.task_id`.
3. Update session runtime tests to assert tool-call `turn_id` and `step_id`.
4. Update runtime control-plane developer docs and zh-CN mirrors where the
   current runtime projections are listed.
5. Validate the session runtime, turn engine, modular IO, and UI gateway
   tool-call paths; compile touched runtime files; grep stale tool-call task
   ownership; run scoped diff checks.
6. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

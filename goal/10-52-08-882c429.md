# 10-52-08 Prompt Context Runtime Plan

## Hermes vs Demiurge comparison

- Hermes 把 system prompt assembly 放在 `agent/prompt_builder.py`，把 context budget/compaction 放在 `agent/context_engine.py` 一类 module 中。即使它的具体内容和 Demiurge 不同，它的 seam 是清楚的：provider request 的上下文构造不是 gateway/platform adapter 的职责，也不应该继续堆在 agent facade。
- 当前 Demiurge 已经有 `ContextAssembler`，但真正的 provider prompt runtime 仍在 runner：
  - `SessionTurnStepRunner._build_messages(...)` 查询 session history、bootstrap context、compaction summary；
  - runner 负责 emit `context.assembled`；
  - runner 负责 `debug.show_system_prompt` 的 delivery、unrouted/failed event、formatting；
  - `TurnEngine` 通过 `RunnerTurnEngineHost` 回调 runner 来拿 provider messages。
- 这让 runner 继续承担 prompt/context lifecycle，且让后续 context engine、prompt injection scan、provider-specific context shaping 很难放到一个稳定 seam。

## Breaking redesign target

新增 `PromptContextRuntime`：

- `PromptContextRuntime.build_messages(...)` 拥有 provider messages 组装和 `context.assembled` event。
- `PromptContextRuntime.deliver_system_prompt_debug(...)` 拥有 debug system prompt delivery。
- `ContextAssembler` 继续是纯组装 implementation。
- `SessionTurnStepRunner` 不再实现 `_build_messages`、`build_turn_messages`、`_maybe_deliver_system_prompt_debug`、`deliver_turn_system_prompt_debug`、`_format_system_prompt_debug`。
- `RunnerTurnEngineHost` 仍通过 runner facade 调用；runner facade 只委托到 prompt context runtime。

Agent Core、Agent Slot、Package 概念不变。本轮不改变 provider payload 语义，只把 ownership 放到正确 runtime seam。

## Deep module check

- Interface size: turn engine 只需要 `build_messages(request)` 和 `deliver_system_prompt_debug(request)`。
- Deletion test: 删除该 module 会把 session history filtering、bootstrap/compaction inclusion、context layer event、system prompt debug formatting、route delivery status handling重新散回 runner。
- Hidden behavior: module 隐藏真实 prompt lifecycle，不是 pass-through。它决定 provider request 的消息层、上下文统计、debug delivery 的跳过/投递/失败事件。
- Adapter discipline: channel/TUI/dashboard 不知道 provider prompt 结构；platform adapter 只接收 debug delivery item。Prompt/context rules 留在 runtime。

## Old interfaces and runner-local behavior to delete

- 删除 runner `_build_messages(...)`。
- 删除 runner `_maybe_deliver_system_prompt_debug(...)`。
- 删除 runner `_format_system_prompt_debug(...)`。
- 删除 runner `_build_skill_index(...)`。
- 将 `build_turn_messages(...)` 和 `deliver_turn_system_prompt_debug(...)` 改成委托。

## New seam placement

- New module: `demiurge/runtime/prompt_context.py`
  - `PromptBuildRequest`
  - `PromptDebugRequest`
  - `PromptContextRuntime`
- Runner composition:
  - runner 持有 `self.prompt_context = PromptContextRuntime(...)`。
  - runtime 接收 `ContextAssembler`、session runtime、event log、interaction router、session id getter、debug flag getter。

## Why this is framework redesign, not ordinary cleanup

- Provider prompt 是 harness 的核心产物。把它从 runner 移出后，新的 Demiurge harness 形成更清楚的链路：turn pipeline -> turn engine -> prompt context runtime -> provider.
- 后续要做 Hermes-style context engine、prompt security scan、provider-specific prompt shaping、context budget tracing，都能在这个 seam 扩展，而不是继续污染 runner。
- 这一步减少 runner 对 provider prompt lifecycle 的 ownership，继续把 runner 压回 composition facade。

## Implementation steps

1. 新增 `demiurge/runtime/prompt_context.py`。
2. 将 provider message build 和 system prompt debug delivery 迁入新 module。
3. 在 runner 初始化 `PromptContextRuntime`，删除本地 context assembler 字段或只作为 runtime dependency。
4. 将 runner 的 `build_turn_messages(...)`、`deliver_turn_system_prompt_debug(...)` 改成委托。
5. 增加 focused tests 覆盖：
   - build messages 排除当前 turn history、包含 bootstrap/compaction、emit `context.assembled`；
   - debug disabled 不投递；
   - debug enabled 且无 route 时 emit unrouted/skip 状态。

## Current Comparison

### Hermes reference strengths

Hermes has several deep modules worth borrowing conceptually, without copying code:

- `gateway/session.py` treats message origin as a first-class record. Its `SessionSource`, deterministic session key, reset/resume flags, and session store concentrate platform session behavior behind one interface.
- `docs/relay-connector-contract.md` and `gateway/relay/*` define a normalized connector handshake and message frame contract. Platform transport details are adapter-owned; gateway routing works on normalized events.
- `gateway/stream_dispatch.py` routes typed stream events through adapter render hooks. The event dispatcher contains no platform knowledge and protects the agent loop from presentation failures.
- `tools/delegate_tool.py` models child agents as isolated execution records with active registry, pause/interrupt controls, depth/concurrency limits, and a restricted toolset.
- `gateway/status.py` records runtime process status and lock semantics explicitly, making operator/dashboard state queryable.
- `agent/turn_context.py`, `agent/conversation_loop.py`, and `agent/turn_finalizer.py` split turn prologue, model/tool loop, and finalization. Demiurge should take the same lesson conceptually: lifecycle phases need explicit modules, not one expanding runner.
- `hermes_cli/plugins.py` and the observability plugins show a host-level hook/middleware seam for runtime telemetry. Demiurge has stronger durable runtime primitives, but still needs a unified observability seam above runner/tool internals.

### Demiurge strengths to preserve

- Agent Core is already file-backed, Git-versioned, and separable from host-owned runtime systems.
- Agent Slots already express authored bootstrap/input/output behavior and package-installed capability shape.
- Package recipes already install reusable authored capability packs without host dependency changes.
- Runtime control plane, session runtime, outbox, and task worker are already moving toward durable host-owned modules.

### Demiurge friction

- `demiurge/runtime/runner.py` still owns too many concepts: session resolution, bootstrap, input pipeline, context assembly, model loop, tool loop, output pipeline, delivery, delegation, state clients, and runtime projections.
- `demiurge/runtime/turn.py` is a shallow module. `TurnEngine` appears separate, but it receives the entire runner and calls private host methods. Its actual Interface is the runner's private implementation.
- `demiurge/runtime/slots.py` only invokes a single handler. The real slot pipeline behavior lives in the runner, so slot scheduling/failure/context behavior has poor locality.
- `demiurge/runtime/interactions.py` normalizes some channel behavior, but adapters still need to understand runner/session details instead of a rich conversation route contract.
- TUI and channel surfaces consume a mix of interaction items, control-plane status, outbox delivery state, and ad-hoc gateway events rather than one typed runtime event stream.
- `RuntimeControlPlane` is valuable, but callers can still bypass its Interface by appending raw events through `RuntimeStore`.
- Background completion claiming and acking is split across TUI, common channel bridge, Telegram bridge, and runner metadata. This should become a dedicated completion inbox module in a later loop.

## Global Redesign Direction

### 1. Host Orchestration Spine

Create a host-owned orchestration spine that is explicit about four phases:

1. Resolve conversation and session.
2. Run authored slot pipelines.
3. Run model/tool loop.
4. Commit runtime events, messages, deliveries, tasks, and observable status.

The external module Interface should be small: "run this normalized inbound turn for this core/session route and return a committed turn result." The implementation can contain many internal seams, but callers should not assemble runtime details themselves.

### 2. Turn Engine as a Deep Module

The model/tool loop should be a deep module behind an explicit host Interface:

- Inputs: core, turn, capability facade, context contributions, available tools, interaction metadata, bootstrap flag.
- Host Interface: build messages, deliver debug prompt, call provider, execute tools, emit events, append runtime events, send interaction progress.
- Outputs: final text, needs-user flag, tool records, model messages, interaction items.

This preserves Agent Core and Agent Slot concepts while making provider/tool-loop correctness testable without constructing a full `SessionTurnStepRunner`.

The first implementation should not stop at replacing `TurnEngine(self)` with a Protocol. The Adapter must depend on explicit runner turn methods and must send runtime event writes through the control-plane seam, otherwise the new Interface is just the old private runner surface in a different file.

### 3. Slot Pipeline Runtime

Move bootstrap/input/output phase execution into a `SlotPipelineRuntime`:

- Input phase returns normalized user text, persisted user text, context contributions, and delivery items.
- Output phase returns delivery items and result mutation.
- Bootstrap phase returns persisted bootstrap context.

The runner supplies host ports for history, state, delivery, skills, tools, and subagents. Agent Slot remains the concept; the execution Interface becomes deeper.

### 4. Conversation Route Contract

Replace loose channel metadata with a typed route contract inspired by Hermes `SessionSource` and relay contract:

- `ConversationOrigin`: channel, conversation key, user identity, thread key, reply target, platform-specific capabilities.
- `ChannelDescriptor`: delivery limits, markdown dialect, draft/edit/thread support, approval support.
- `ConversationRoute`: binds normalized inbound and outbound delivery.

Channel adapters become adapters at this seam. Session binding becomes host-owned and testable through normalized origins.

### 5. Runtime Event Stream and Dashboard

Unify runtime observability through typed event envelopes:

- turn events
- model step events
- tool call events
- slot events
- delivery events
- task events
- approval/prompt events

Persistence, channel delivery, TUI, and dashboard should be projections of this stream, not independent partial interpretations.

### 6. UI Gateway Protocol and Transport

The local TUI should stop treating the Python child process protocol as loose JSON. Define a versioned UI gateway protocol with typed methods, events, error envelopes, and transport adapters:

- stdio adapter for the current TUI.
- WebSocket adapter for future dashboard and remote UI.
- tee/debug adapter for recording UI protocol traces in tests.

This mirrors Hermes' split between gateway runtime, stream events, relay transport, and web/dashboard clients.

### 7. Ingress Turn Controller

Busy-mode queueing, interrupt semantics, pending prompt replies, and background completion merging currently live inside specific bridge adapters. Introduce an ingress controller that receives normalized inbound turns and owns:

- interrupt vs queue policy
- pending prompt/approval routing
- task completion claim/merge
- run-id/turn-id handoff to the host orchestration spine

TUI, Telegram, webhook, and future dashboard adapters should become thin adapters around this Interface.

### 8. Channel Capability Descriptor

Extend channel registration with an explicit descriptor, inspired by Hermes relay descriptors:

- message length and markdown dialect
- rich block support
- edit/draft/stream support
- approval and prompt support
- schedule target support
- auth ownership and user/session isolation hints

This gives the Host enough information to project runtime events and deliveries without channel-specific conditionals leaking upward.

### 9. Package Capability Transactions

Keep Package capability packs as authored capabilities, but express installation as a typed transaction:

- files to write
- pipeline changes
- config changes
- provenance expectations
- validation hooks

This makes package install/uninstall auditable, reversible, and easier for future Agent Core evolution to reason about.

## This Loop Implementation Scope

This loop will implement the first deepening candidate only:

> Make `TurnEngine` depend on an explicit `TurnEngineHost` Interface plus a runner Adapter, instead of depending on `SessionTurnStepRunner` as an implicit private-method surface.

### Why this first

- It is the smallest code change that starts the redesign direction.
- It does not alter the concepts of Agent Core, Agent Slot, or Package capability packs.
- It reduces the runner as god module without changing user-visible behavior.
- It gives future refactors a stable seam for replacing the model/tool loop.
- It can be verified with focused runtime tests.

### Planned code changes

- Add `TurnEngineHost` Protocol and `RunnerTurnEngineHost` Adapter in `demiurge/runtime/turn.py`.
- Change `TurnEngine` to call the explicit host Interface instead of `self.host._private_method` and direct runner attributes.
- Update `SessionTurnStepRunner` to construct `TurnEngine(RunnerTurnEngineHost(self))`.
- Add turn-oriented runner methods for message assembly, debug delivery, provider calls, tool execution, progress delivery, output clients, event emission, and runtime-event append.
- Add `RuntimeControlPlane.record_events()` and route turn-engine runtime event writes through that seam.
- Add focused tests for `TurnEngine` using a fake host, covering:
  - direct final assistant response
  - tool-call step followed by final response
  - terminating tool result with `needs_user`
- Run targeted tests:
  - `uv run pytest tests/runtime/test_turn_engine.py`
  - `uv run pytest tests/runtime/test_harness.py tests/runtime/test_session_runtime.py`
  - `uv run python -m compileall demiurge/runtime tests/runtime`

### Out of scope for this loop

- Moving slot pipeline execution out of `runner.py`.
- Rewriting channel adapters or TUI.
- Changing public CLI/docs behavior.
- Changing Agent Core schema, Agent Slot shape, or package recipe shape.
- Copying Hermes implementation code.

# 13-00-05 Agent Slot Phase Selection Plan

## Global architecture review

The previous loops made slot execution and child-agent invocation deeper
runtime modules. The next framework-shape problem is Agent Slot phase selection:

- `ResolvedPhaseSlots` lives in `demiurge/runtime/child_agents.py`, even though
  it is a generic Agent Slot pipeline concept used by turn execution too.
- `SessionTurnStepRunner._run_input_slots(...)` and `_run_output_slots(...)`
  only choose serial/parallel phase slots and forward to `SlotPipelineRuntime`.
- `TurnPipelineRuntime` depends on the child-agent module just to describe a
  slot phase override.

This is a wrong seam and a shallow runner facade. Child-agent runtime should
use the slot phase type; it should not own the type. Runner should not know the
rules for choosing core pipeline slots versus explicit serial overrides versus
fully resolved phase overrides.

## Hermes vs Demiurge comparison

- Hermes keeps conversation-loop phase metadata close to the loop/harness
  rather than making a subagent helper own a generic phase-selection type.
- Demiurge's Agent Slot concept is stronger than Hermes' helper modules, so the
  right owner is `SlotPipelineRuntime`: it is the module that already owns
  phase scheduling, per-slot execution, background flushing, skill activation,
  and slot effects.
- The useful Hermes lesson is locality: runtime phase metadata should travel
  with the lifecycle that consumes it, not through an unrelated module.

## Breaking redesign target

- Move `ResolvedPhaseSlots` from `child_agents.py` to `slots.py`.
- Let `SlotPipelineRuntime` resolve input/output serial and parallel slots from:
  - a full `ResolvedPhaseSlots` override;
  - explicit serial-only overrides from CLI/child/delegation surfaces;
  - the core's default phase pipeline.
- Delete `SessionTurnStepRunner._run_input_slots(...)`.
- Delete `SessionTurnStepRunner._run_output_slots(...)`.
- Change `TurnPipelineHost` so it passes `InputPipelineRequest` /
  `OutputPipelineRequest` objects to the slot runtime instead of leaking
  lifecycle internals and relying on runner wrappers.

Agent Core, Agent Slot, and Package concepts stay unchanged. Authored
`agent/pipelines.yaml`, input/output slots, child slot selection, and
`ctx.agents.*` behavior remain the same; this only changes host-internal
runtime ownership.

## Deep module check

- Interface size: turn pipeline asks the slot runtime to run an input or output
  phase request; it no longer has to know the selection rule in runner.
- Deletion test: deleting this slot phase selection logic would scatter rules
  back into runner, child-agent runtime, and turn pipeline tests.
- Hidden behavior: `SlotPipelineRuntime` hides real behavior: default pipeline
  selection, explicit override semantics, serial/parallel scheduling, per-slot
  execution, effects, and failure policy.
- Adapter discipline: runner remains composition/root wiring; child-agent
  runtime creates slot phase overrides but no longer owns the generic phase
  type.

## Old interfaces and files to delete

- Delete `ResolvedPhaseSlots` from `demiurge/runtime/child_agents.py`.
- Delete runner methods:
  - `_run_input_slots(...)`
  - `_run_output_slots(...)`
- Remove `InputPipelineRequest` / `OutputPipelineRequest` imports from runner.
- Remove turn pipeline's dependency on `demiurge.runtime.child_agents` for
  `ResolvedPhaseSlots`.

## New seam placement

- `demiurge/runtime/slots.py`
  - owns `ResolvedPhaseSlots`;
  - owns phase selection helpers for input/output;
  - accepts optional serial overrides and full resolved phase overrides in
    pipeline requests.
- `demiurge/runtime/child_agents.py`
  - imports `ResolvedPhaseSlots` from `slots.py`;
  - keeps child-agent slot selection behavior but no longer owns the shared
    type.
- `demiurge/runtime/turn_pipeline.py`
  - constructs slot pipeline request objects from turn lifecycle data;
  - calls host methods that directly run those request objects.
- `demiurge/runtime/runner.py`
  - wires `SlotPipelineRuntime` only;
  - no longer exposes input/output slot pipeline wrapper methods.

## Why this is framework redesign, not ordinary cleanup

This removes a false ownership signal where child-agent runtime appeared to own
generic Agent Slot phase data. It also removes another runner pass-through loop.
The resulting shape is closer to a new Demiurge framework: Agent Slot lifecycle
semantics are concentrated in a deep slot runtime, while child-agent and turn
lifecycles produce requests for that runtime.

## Implementation steps

1. Move `ResolvedPhaseSlots` to `slots.py`.
2. Extend `InputPipelineRequest` and `OutputPipelineRequest` with
   `phase_slots` and optional `serial_slots`/`parallel_slots`.
3. Add slot phase selection helpers inside `SlotPipelineRuntime`.
4. Update child-agent, runner, and turn pipeline imports.
5. Change `TurnPipelineHost.run_input_slots/run_output_slots` to accept
   pipeline request objects and call `slot_pipeline` directly.
6. Delete runner `_run_input_slots` and `_run_output_slots`.
7. Update tests that import `ResolvedPhaseSlots` or assert turn host request
   shapes.

# 13-13-26 Turn Engine IO Facade Collapse Plan

## Global architecture review

The previous loop moved Agent Slot phase selection into `SlotPipelineRuntime`.
The next shallow seam is the Turn Engine IO facade:

- `TurnIO` already owns assistant-step persistence, tool-result persistence,
  tool start/finish interaction items, and channel dispatch.
- `SlotContextRuntime` already owns module output clients for tool effects.
- `RunnerTurnEngineHost` still calls runner wrapper methods:
  `send_turn_assistant_step(...)`, `send_turn_tool_call_started(...)`,
  `send_turn_tool_call_finished(...)`, and `turn_output_client(...)`.
- Those runner methods only forward to `TurnIO` or `SlotContextRuntime`.

This keeps runner in the model/tool loop even though runner is not the module
that varies. The turn engine adapter should connect directly to runtime modules
that own the behavior.

## Hermes vs Demiurge comparison

- Hermes' decomposed conversation loop keeps model-step IO behavior close to
  the conversation-loop helpers, not behind an extra root-object forwarding
  layer.
- Demiurge should keep `TurnEngine` as the model/tool loop and `TurnIO` as the
  transcript/delivery module. Runner should compose those modules, not proxy
  every IO call.
- The useful Hermes lesson is to avoid god-object tails: once behavior has a
  runtime module, callers should use that module at the seam.

## Breaking redesign target

- Delete runner wrapper methods:
  - `_module_io_client(...)`
  - `turn_output_client(...)`
  - `send_turn_assistant_step(...)`
  - `send_turn_tool_call_started(...)`
  - `send_turn_tool_call_finished(...)`
- Update `RunnerTurnEngineHost` to call:
  - `runner.runtime_io.send_assistant_step(...)`;
  - `runner.runtime_io.send_tool_call_started(...)`;
  - `runner.runtime_io.send_tool_call_finished(...)`;
  - `runner.slot_context.module_io_client(...)` for tool output clients.

Agent Core, Agent Slot, and Package concepts stay unchanged. Tool execution,
delivery, history, and channel-visible tool lifecycle behavior should be
identical.

## Deep module check

- Interface size: `TurnEngineHost` keeps the model/tool loop interface, while
  runner loses methods that were only forwarding implementation details.
- Deletion test: deleting the runner wrappers does not scatter behavior; the
  behavior already lives in `TurnIO` and `SlotContextRuntime`. That proves the
  wrappers are shallow and should be removed.
- Hidden behavior: `TurnIO` remains the deep module hiding transcript and
  interaction-item details; `SlotContextRuntime` remains the deep module hiding
  module IO clients.
- Adapter discipline: `RunnerTurnEngineHost` is the adapter from turn engine to
  runtime modules. Runner is composition/root wiring only.

## Old interfaces to delete

- `SessionTurnStepRunner._module_io_client(...)`
- `SessionTurnStepRunner.turn_output_client(...)`
- `SessionTurnStepRunner.send_turn_assistant_step(...)`
- `SessionTurnStepRunner.send_turn_tool_call_started(...)`
- `SessionTurnStepRunner.send_turn_tool_call_finished(...)`

## New seam placement

- `demiurge/runtime/turn.py`
  - `RunnerTurnEngineHost` directly calls runtime modules already held by the
    runner.
- `demiurge/runtime/runner.py`
  - only wires `TurnIO`, `SlotContextRuntime`, and `TurnEngine`;
  - no longer exposes turn-engine IO forwarding methods.

## Why this is framework redesign, not ordinary cleanup

This removes another runner-mediated loop in the heart of the model/tool
execution path. The resulting shape is closer to the desired framework:
runtime modules own lifecycle behavior, adapters connect modules, and runner is
the composition root.

## Implementation steps

1. Update `RunnerTurnEngineHost` methods to call runtime modules directly.
2. Delete the runner IO forwarding methods.
3. Remove stale runner imports.
4. Run turn-engine, turn-pipeline, and modular-IO tests to verify no visible IO
   behavior changed.

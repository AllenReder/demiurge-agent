# Background Work Runtime Extraction Plan

## Hermes vs Demiurge Comparison

Hermes treats background execution as a managed runtime concern. Its process
registry owns in-memory process handles, drain/wait/kill behavior, output
buffers, and crash recovery; gateway and TUI code attach route or subscription
metadata but do not own the background lifecycle itself.

Demiurge already has `RuntimeTaskWorker` for durable background tasks, but
`SessionTurnStepRunner` still owns a separate `_background_tasks` set for
in-process delivery and slot coroutines. It also exposes `track_slot_background_task`,
`drain_background_tasks`, and `background_task_count`, so interaction runtime,
child-agent runtime, scheduler, TUI status, slot pipeline, and delivery dispatch
all couple to runner as the background-work module.

## Framework Shape Decision

Background work should be a runtime module, not runner state:

- `BackgroundWorkRuntime` owns local async task tracking.
- It composes with `RuntimeTaskWorker` for optional durable task draining and
  active counts.
- Slot pipelines and interaction delivery dispatch track local background work
  through the background-work module.
- Runners keep a `background_tasks` runtime object but no longer expose
  background lifecycle methods or a private `_background_tasks` set.

This keeps foreground turn orchestration and background lifecycle ownership at
separate seams.

## Delete Or Merge

- Delete `SessionTurnStepRunner._background_tasks`.
- Delete `SessionTurnStepRunner.track_slot_background_task`.
- Delete `SessionTurnStepRunner.drain_background_tasks(...)`.
- Delete `SessionTurnStepRunner.background_task_count`.
- Replace call sites with `runner.background_tasks.track(...)`,
  `runner.background_tasks.drain(...)`, and
  `runner.background_tasks.active_count`.

## New Seam

Add `demiurge.runtime.background.BackgroundWorkRuntime`:

- `track(task)` records an in-process background coroutine and removes it when
  it finishes.
- `drain(include_runtime_tasks=True)` drains tracked local work and optionally
  drains the durable `RuntimeTaskWorker`.
- `active_count` returns local active coroutine count plus durable task-worker
  active count.

The module hides the difference between local background coroutines and durable
runtime tasks from callers that only need drain/status behavior.

## Ownership After This Change

- Runtime module: local background coroutine tracking, optional durable task
  drain, active background work count.
- Runner: wires runtime modules together and runs foreground turns.
- Adapters: render status from the background-work interface, but do not know
  runner internals.
- Authored Agent Core / Agent Slot / Package: concepts unchanged; slots can
  still schedule parallel work, and package tools can still create durable
  runtime tasks through host-controlled capabilities.

## Why This Is A Demiurge Framework Step

This is not ordinary helper extraction. It removes a host-lifecycle concern
from the runner monolith and creates a smaller, deeper module interface for all
background work observation and draining. Deleting the old runner fields would
otherwise scatter the behavior across slot runtime, delivery dispatch,
interaction runtime, child-agent runtime, scheduler, and UI status code.

## Implementation Steps

1. Add `BackgroundWorkRuntime` with unit tests for track, drain, and active
   count composition with a fake task worker.
2. Wire `SessionTurnStepRunner.background_tasks` during initialization.
3. Pass `background_tasks.track` into `SlotPipelineRuntime` and
   `InteractionDispatchRuntime`.
4. Replace internal and test call sites that drained via runner with
   `runner.background_tasks.drain(...)`.
5. Replace UI status count with `runner.background_tasks.active_count`.
6. Run targeted background, modular IO, interaction, scheduler, package-memory,
   and channel tests; compile touched files; run scoped diff checks.
7. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

# 12-11-47 Conversation Lifecycle Primitive Collapse Plan

## Global architecture review

After the channel and TUI migrations, `ConversationLifecycleRuntime` is the
runtime seam for adapter conversation lifecycle. Two helper modules are now
shallow:

- `demiurge/runtime/conversation_states.py` is only imported by
  `conversation_lifecycle.py` and its own tests.
- `demiurge/runtime/completion_delivery.py` is only imported by
  `conversation_lifecycle.py` and its own tests.

Their public interfaces are nearly the same complexity as their implementation.
Keeping them separate makes the new lifecycle module look like a coordinator of
helper modules instead of the owner of lifecycle behavior.

## Hermes vs Demiurge comparison

- Hermes gateway lifecycle code keeps active-session state, busy queues, and
  pending follow-up continuation together where the gateway lifecycle decision
  is made.
- Hermes does not require each platform adapter to understand a stack of small
  lifecycle helper modules.
- Current Demiurge has reached the same gateway/runtime shape, but still
  exposes micro-modules that only exist because the refactor happened in
  multiple incremental rounds.

## Breaking redesign target

Collapse the lifecycle primitives into `demiurge/runtime/conversation_lifecycle.py`:

- Move conversation state caching and task-worker subscription ownership into
  the lifecycle module.
- Move pending completion merge/enqueue ownership into the lifecycle module.
- Delete `conversation_states.py` and `completion_delivery.py`.
- Move any remaining valuable tests into
  `tests/runtime/test_conversation_lifecycle.py`.

This keeps Agent Core, Agent Slot, and Package concepts unchanged. It changes
only host runtime internals.

## Deep module check

- Interface size: callers keep using `ConversationLifecycleRuntime`; they no
  longer need to know `ConversationStateStore` or `CompletionDeliveryRuntime`
  exist.
- Deletion test: deleting the two helper modules concentrates behavior in the
  lifecycle module instead of scattering it. Deleting
  `ConversationLifecycleRuntime` would still scatter behavior across Text,
  Telegram, and TUI adapters, so it remains the deep module.
- Hidden behavior: state lookup, task-worker subscription, completion claiming,
  pending merge, enqueue, finish, drain, and adapter hooks live behind one
  runtime interface.
- Adapter discipline: adapters remain unchanged in concept. This pass does not
  move transport/UI behavior into runtime.

## Old interfaces and files to delete

- Delete `demiurge/runtime/conversation_states.py`.
- Delete `demiurge/runtime/completion_delivery.py`.
- Delete their dedicated tests after moving important coverage into
  `tests/runtime/test_conversation_lifecycle.py`.
- Remove imports of the deleted modules from `conversation_lifecycle.py`.

## New seam placement

- `demiurge/runtime/conversation_lifecycle.py`
  - owns `ConversationLifecycleRuntime`;
  - owns private state cache/subscription implementation;
  - owns private completion merge/enqueue implementation;
  - continues to use `ConversationIngressState` and `ConversationTurnController`
    from `ingress.py` for queue and active-task primitives.

## Why this is framework redesign, not ordinary cleanup

- It prevents the new architecture from becoming a stack of shallow helpers.
- It makes the lifecycle module the actual host-owned conversation lifecycle
  seam for every adapter.
- It reduces the number of runtime modules future task/session changes must
  understand.

## Implementation steps

1. Inline the state store implementation into `conversation_lifecycle.py`.
2. Inline the completion delivery implementation into `conversation_lifecycle.py`.
3. Move coverage from deleted helper tests into
   `tests/runtime/test_conversation_lifecycle.py`.
4. Delete the two helper modules and tests.
5. Run focused runtime and adapter lifecycle tests.

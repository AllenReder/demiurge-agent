## Reference Comparison

### Hermes reference

Hermes keeps conversation origin and route facts in named records:

- `gateway/session.py` defines `SessionSource`, which packages platform, chat, user, thread, message, and profile facts.
- `gateway/delivery.py` resolves delivery targets from explicit targets or origin records rather than requiring each adapter to rebuild the same route facts.

The transferable lesson is not Hermes's exact schema. Demiurge should first name its shared ingress state so adapters stop duplicating route memory and completion-route construction.

### Demiurge current state

This branch now has:

- `TurnEngine` for model/tool loop orchestration.
- `SlotPipelineRuntime` for input/output phase scheduling.
- `SlotContextRuntime` for authored `ctx.*` construction.
- `TurnIO` for model-step transcript and interaction-item assembly.
- `ModuleDeliveryRuntime` for authored delivery request application.
- `ChildAgentRuntime` for child-agent lifecycle.
- `CompletionInbox` for background completion claim, inbound construction, merge metadata, and ack.
- `InboundQueueRuntime` for inbound queue selection, completion merge, and completion-preserving clear.

The remaining shallow cluster is conversation route state:

- `TextConversationState` and `TelegramConversationState` duplicate `runtime`, `busy_mode`, `route_binding`, `conversation_key`, `source`, `reply_to`, `metadata`, `active_task`, and `queue`.
- `TextChannelBridgeBase._remember_route(...)` and `TelegramInteractionBridge._remember_route(...)` are identical.
- `TextChannelBridgeBase._merge_stored_task_completions(...)` and `TelegramInteractionBridge._merge_stored_task_completions(...)` rebuild the same session/task-worker lookup and `CompletionRoute` shape.
- `_state_for_session(...)` in both adapters scans conversation state through the same runner/session lookup.

Deletion test: deleting either adapter's state and route helpers would force the same route-memory and completion-route logic to reappear. Two real adapters justify the seam.

## This Loop Scope

Extract a `ConversationIngressState` Module that owns shared ingress state fields, route memory, session lookup, completion route construction, and pending completion claims.

This is behavior-preserving. It changes internal Module placement and adapter call sites only; it does not change Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, task tool semantics, completion event schema, busy mode meanings, prompt behavior, approval behavior, command behavior, TUI rendering, or Telegram delivery behavior.

## Planned Code Changes

- Extend `demiurge/runtime/ingress.py` with `ConversationIngressState`.
- Move shared fields into the Module:
  - `runtime`;
  - `busy_mode`;
  - `route_binding`;
  - `conversation_key`;
  - `source`;
  - `reply_to`;
  - `metadata`;
  - `active_task`;
  - `queue`.
- Add `ConversationIngressState` methods/properties:
  - `remember_route(inbound)`;
  - `runner`;
  - `session_id`;
  - `task_worker`;
  - `completion_route(channel, fallback_source=...)`;
  - `claim_pending_completions(channel, owner_id, fallback_source=...)`;
  - `claim_completion_event(event, channel, owner_id, task_worker=...)`.
- Update `TextChannelBridgeBase`:
  - replace local `TextConversationState` dataclass with the shared state Module;
  - delegate route memory to `state.remember_route(...)`;
  - delegate stored completion claim/route construction to `state.claim_pending_completions(...)`;
  - delegate subscription completion claim/route construction to `state.claim_completion_event(...)`;
  - use `state.session_id` in `_state_for_session(...)`.
- Update `TelegramInteractionBridge`:
  - make `TelegramConversationState` extend `ConversationIngressState` and keep only Telegram-specific `pending_approval_id`;
  - delegate route memory and stored completion route construction to the shared Module;
  - delegate subscription completion claim/route construction to the shared Module;
  - use `state.session_id` in `_state_for_session(...)`.
- Add focused tests in `tests/runtime/test_ingress.py`:
  - `remember_route(...)` copies source, reply target, conversation key, and metadata;
  - `completion_route(...)` uses remembered route facts and fallback source;
  - `session_id` reads the active runner session;
  - `claim_pending_completions(...)` claims pending completion events using the shared route;
  - `claim_completion_event(...)` claims a single subscribed completion event using the shared route.

## Expected Benefits

- Conversation route state becomes one deep Module Interface instead of two copied dataclasses and helper clusters.
- Locality improves: route memory, session lookup, completion routes, and queue state live together.
- Leverage improves: text channels and Telegram reuse the same route-state Module.
- The later `IngressTurnController` can depend on a named ingress-state Module instead of adapter-private dataclasses.

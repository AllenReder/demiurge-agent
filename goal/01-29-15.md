## Reference Comparison

### Hermes reference

Hermes continues to be useful as a lifecycle-decomposition reference, not as code to copy.

- `agent/turn_context.py` names a prologue Module that produces a compact turn product.
- `agent/turn_finalizer.py` names a post-loop Module that consumes a compact request-like set of values.
- `agent/conversation_loop.py` is easier to reason about because lifecycle details are moved behind named Interfaces.

The transferable lesson is that a deep Module should carry phase-local facts through a small Interface. Demiurge's Agent Slot model is different, so the redesign should deepen Demiurge's slot seam instead of copying Hermes.

### Demiurge current state

The previous loop extracted `SlotPipelineRuntime`, which now owns input/output phase ordering, parallel scheduling, background flushing, and phase result assembly.

The remaining friction is that `SlotPipelineHost` is still shallow:

- `run_input_slot(...)` has a long parameter list exposing builder mutability, raw input, state stores, activation state, contribution lists, metadata, and background mode.
- `run_output_slot(...)` exposes current output, tool records, state stores, result client, metadata, and background mode.
- The Interface is almost the implementation, so the next `SlotContextRuntime` extraction would have to preserve a wide host seam.

Explorer feedback confirmed the next high-leverage direction is a Slot Invocation Context Runtime that owns authored `ctx` construction. This loop is the seam cleanup required before moving that context construction: collapse per-slot execution locals into explicit request Modules.

## This Loop Scope

Implement the next deepening candidate:

> Collapse slot execution requests so `SlotPipelineHost` accepts one input slot request or one output slot request instead of leaking every phase-local variable across the seam.

This keeps behavior unchanged while making the slot execution seam smaller and more testable.

## Planned Code Changes

- Add `InputSlotRunRequest` and `OutputSlotRunRequest` dataclasses in `demiurge/runtime/slots.py`.
- Change `SlotPipelineHost.run_input_slot(...)` and `run_output_slot(...)` to accept one request object each.
- Change `SlotPipelineRuntime` to construct those request objects for serial and parallel slot execution.
- Change `RunnerSlotPipelineHost` and `SessionTurnStepRunner.run_input_pipeline_slot(...)` / `run_output_pipeline_slot(...)` to consume request objects.
- Keep `_run_input_slot(...)` and `_run_output_slot(...)` as the actual runner-owned invocation implementation for now.
- Update `tests/runtime/test_slot_pipeline.py` so the fake host asserts request object invariants instead of matching a long parameter list.

## Out Of Scope

- Moving `ModuleInputClient`, `ModuleOutputClient`, `ModuleHistoryClient`, `ModuleStateClient`, `ModuleAgentsClient`, `ModuleToolClient`, `ModuleSkillClient`, `ModuleResultClient`, or `ModuleIOClient` out of `runner.py`.
- Moving `InputContext` / `OutputContext` construction out of `SessionTurnStepRunner`.
- Moving bootstrap execution.
- Changing Agent Core file layout, slot YAML, pipelines YAML, or package recipes.
- Changing channel/bridge/TUI delivery behavior.
- Copying Hermes implementation code.

## Expected Benefits

- `SlotPipelineHost` becomes a smaller Interface.
- The slot pipeline Module gains locality: phase scheduling creates the exact request consumed by the host Adapter.
- The future `SlotContextRuntime` extraction has one request shape to accept instead of duplicating a long host parameter list.
- Tests exercise request invariants through the pipeline seam.

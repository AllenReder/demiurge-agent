# 12-17-25 Channel Command Lifecycle Interface Plan

## Global architecture review

`ConversationLifecycleRuntime` now owns channel and TUI conversation lifecycle.
The remaining non-test caller of `ConversationTurnController` is
`demiurge/channels/commands.py`.

That means slash command handling still knows the low-level lifecycle primitive
for:

- running-state inspection for `/status`;
- queue clearing for `/new` and `/stop`;
- drain-next after `/stop`;
- queue-and-drain for `/queue`.

This is the last adapter-facing leak of the low-level turn controller.

## Hermes vs Demiurge comparison

- Hermes gateway command/control paths operate against the gateway lifecycle
  state, not a separate low-level queue controller exposed to platform command
  handlers.
- Current Demiurge has the right lifecycle runtime, but channel command handlers
  still bypass it for command-specific lifecycle control.
- The adapter split should be: commands decide user intent; lifecycle runtime
  executes queue/stop/drain/running semantics.

## Breaking redesign target

Add a small command-control interface to `ConversationLifecycleRuntime`:

- `queued_count(state)`
- `clear_queue(state, preserve_completions=...)`
- `queue_and_drain_if_idle(state, inbound)`
- `drain_next(state)`

Then migrate `ChannelCommandExecutor` to use this lifecycle interface for
status/new/stop/queue instead of importing `ConversationTurnController`.

Telegram can still pass its custom `cancel_active` callback so approval cleanup
remains platform-specific. Text channel uses the lifecycle default cancel path.

## Deep module check

- Interface size: channel commands only learn lifecycle-level verbs, not the
  lower-level controller and run callback wiring.
- Deletion test: deleting the lifecycle command-control methods would push
  queue/drain/cancel wiring back into command handlers and adapters.
- Hidden behavior: lifecycle runtime owns how queued inputs become turns and
  how drain uses the lifecycle turn wrapper.
- Adapter discipline: command parsing/output stays in `channels/commands.py`;
  platform-specific cancellation hooks stay in the adapter.

## Old interfaces and behavior to delete

- Delete `ConversationTurnController` import from `demiurge/channels/commands.py`.
- Delete command-local drain callback wiring.
- Replace default command cancellation with lifecycle cancellation.

## New seam placement

- `demiurge/runtime/conversation_lifecycle.py`
  - exposes lifecycle command-control methods.
- `demiurge/channels/commands.py`
  - depends on the lifecycle interface rather than the low-level controller.
- `demiurge/channels/base.py` and `demiurge/channels/telegram/bridge.py`
  - construct lifecycle runtime before `ChannelCommandExecutor` and pass it in.

## Why this is framework redesign, not ordinary cleanup

- It removes the last adapter-facing reference to the low-level turn controller.
- It makes the lifecycle runtime the single host-owned module for conversation
  state, queueing, stopping, and drain semantics across channel adapters.
- It prepares a later pass where `ingress.py` can become a private primitive
  module tested mostly through lifecycle behavior.

## Implementation steps

1. Add command-control methods to `ConversationLifecycleRuntime`.
2. Update `ChannelCommandExecutor` to require and use a lifecycle runtime.
3. Reorder Text and Telegram bridge initialization so command executors receive
   the lifecycle runtime.
4. Run channel command, text/channel, Telegram, and lifecycle tests.

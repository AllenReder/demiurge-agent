## Reference Comparison

### Hermes reference

Hermes continues to be useful as a lifecycle naming reference:

- `agent/turn_context.py` packages turn prologue work into a named product consumed by the loop.
- `agent/turn_finalizer.py` packages post-loop work into a named product consumed by the caller.
- `agent/conversation_loop.py` stays more navigable because repeated transcript/tool lifecycle details are not all inline.

The transferable lesson for Demiurge is to name turn lifecycle Modules around Demiurge's own host-owned concepts, not to copy Hermes implementation code.

### Demiurge current state

This branch now has:

- `TurnEngine` behind `TurnEngineHost`.
- `SlotPipelineRuntime` behind `SlotPipelineHost`.
- `SlotContextRuntime` behind `SlotContextHost`.

The next shallow Module is `RuntimeIO`: it is already named, but it still lives inside `demiurge/runtime/runner.py` and calls runner private helpers directly. That hides the seam and keeps model-step transcript persistence, tool-call item construction, and module-output delivery assembly visually inside the runner.

Deletion test: deleting the nested `RuntimeIO` class would force message persistence metadata and `InteractionItem` construction to reappear across the runner's turn-loop methods. That means it earns a real Module seam, but the seam is currently buried in the wrong file.

## This Loop Scope

Extract a `TurnIO` runtime Module from the nested `RuntimeIO` class.

This is behavior-preserving. It changes internal Module placement and the host Adapter shape only; it does not change Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, delivery semantics, or channel behavior.

## Planned Code Changes

- Add `demiurge/runtime/io.py`.
- Move `RuntimeIO` out of `demiurge/runtime/runner.py` and rename it to `TurnIO`.
- Add a `TurnIOHost` Protocol for the host effects the Module actually needs:
  - current `session_id`;
  - `SessionRuntime`;
  - `EventLog`-like `emit`;
  - model-content truncation;
  - tool-result model content formatting;
  - scheduling an `InteractionItem`;
  - dispatching an item immediately;
  - applying a module `DeliveryRequest`.
- Add `RunnerTurnIOHost` as the Adapter from `SessionTurnStepRunner` to `TurnIOHost`.
- Instantiate `self.runtime_io = TurnIO(RunnerTurnIOHost(self))`.
- Keep these concerns host-owned for this loop:
  - delivery request normalization and outbox enqueue;
  - background task tracking;
  - interaction router and bridge dispatch;
  - child-agent run/spawn/delegation;
  - turn engine and slot pipeline orchestration.
- Keep thin compatibility methods in runner where existing code already calls `send_turn_*`.
- Add focused tests in `tests/runtime/test_io.py` for the extracted Module Interface:
  - `send_user` persists user message and emits `message.persisted`;
  - `send_assistant_step` persists assistant step and schedules visible interim delivery only when content is non-empty;
  - `send_tool_result` persists hidden tool result and schedules a tool-result item;
  - `send_tool_call_started` / `send_tool_call_finished` dispatch tool-call lifecycle items;
  - `send_module_output` delegates delivery request application and wraps visible delivery as an `InteractionItem`.

## Out Of Scope

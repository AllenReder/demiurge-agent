# 10-29-50 Turn Pipeline Runtime Plan

## Hermes vs Demiurge comparison

- Hermes 的 gateway/platform adapter 只把外部事件转成 agent session 的一次运行；真正的 session/thread/job lifecycle 在 host runtime 中集中推进。
- Hermes 的可取点不是它的具体 `AIAgent` 类，而是 controller 形状：平台 adapter 不拥有 turn 规则，agent facade 也不应该同时知道 input slot、provider loop、tool loop、output slot、task completion、delivery projection 的精确顺序。
- 当前 Demiurge 已经有 `TurnEngine` 和 `TurnLifecycleRuntime`，但 `SessionTurnStepRunner.run_turn(...)` 仍然是一个 180 行左右的 host lifecycle monolith：
  - 它加载/绑定 core 和 session；
  - 它解析 channel interaction metadata；
  - 它执行 bootstrap、input slot、provider/tool loop、output slot；
  - 它负责 cancelled/failed interrupt；
  - 它更新 session history 和 display projection；
  - 它完成 task ledger 和 turn store。

## Breaking redesign target

把完整 turn pipeline 从 `SessionTurnStepRunner` 移到新的深 module：

- `TurnPipelineRuntime` 拥有一次 agent turn 的端到端 host pipeline。
- `SessionTurnStepRunner.run_turn(...)` 退化为 facade，只把 authored/channel override 组装成 `TurnPipelineRequest` 并调用 runtime。
- `TurnEngine` 继续只拥有 provider/tool step loop。
- `TurnLifecycleRuntime` 继续只拥有 turn/task/session lifecycle projection。
- `SlotPipelineRuntime` 继续只拥有 input/output slot phase execution。

Agent Core、Agent Slot、Package 的概念不变。本轮不改作者可见 schema；只改变 host runtime 的 seam。

## Deep module check

- Interface size: caller 只需要 `TurnPipelineRuntime.run(request)`。request 暴露的是一次 turn 的外部变量：raw text、core override、interaction、injected system context、slot overrides、bootstrap flag、route binding。
- Deletion test: 删除这个 module 会把 core/session binding、bootstrap、slot ordering、tool preparation、engine dispatch、output delivery、history refresh、task completion、interrupt handling重新散回 runner。
- Hidden behavior: module 隐藏真实行为，不是 pass-through。它决定一次 turn 的控制流、异常状态转换、哪些结果进入 `TurnResult`、哪些 delivery 进入 visible history projection。
- Adapter discipline: channel/TUI/dashboard 仍只调用 runner facade；平台 adapter 不知道 input/output slot、provider step 或 task ledger 顺序。Runner 也不再是所有 lifecycle 规则的 owner。

## Old interfaces and runner-local behavior to delete

- 删除 `SessionTurnStepRunner.run_turn(...)` 中的完整生命周期编排。
- 删除 runner 对 `TurnEngineRequest`、`TurnLifecycleRequest`、`TurnLifecycleCompletion` 的直接组装职责。
- 将 `TurnResult` 移出 runner，让 turn pipeline module 成为结果类型 owner。
- 保留 runner 的底层 host adapter 方法作为过渡 seam；下一轮可以继续把过宽的 runner host interface 合并进更深的 runtime owner。

## New seam placement

- New module: `demiurge/runtime/turn_pipeline.py`
  - `TurnResult`
  - `TurnPipelineRequest`
  - `TurnPipelineHost`
  - `RunnerTurnPipelineHost`
  - `TurnPipelineRuntime`
- Runner composition:
  - `SessionTurnStepRunner.__init__` 创建 `self.turn_pipeline = TurnPipelineRuntime(RunnerTurnPipelineHost(self))`
  - `SessionTurnStepRunner.run_turn(...)` 只创建 request 并委托。

## Why this is framework redesign, not ordinary cleanup

- 这是 harness shape 的改变：agent facade 不再拥有 turn lifecycle 的顺序规则。
- 后续 task/job/schedule/session/channel/TUI 都可以围绕同一个 `TurnPipelineRuntime` 交互，而不是继续向 runner 添加新分支。
- Hermes 参考的是“平台 adapter 调度 host runtime，host runtime 拥有 lifecycle”的概念；Demiurge 保留 Agent Core/Slot/Package，但把 host pipeline 从 legacy runner 中抽离。

## Implementation steps

1. 新增 `demiurge/runtime/turn_pipeline.py`，复制并收敛当前 `run_turn` 端到端逻辑。
2. 将 `TurnResult` 移入新 module，并从 runner 重新导入以保留 Python import 兼容。
3. 在 runner 初始化 `TurnPipelineRuntime`。
4. 将 `SessionTurnStepRunner.run_turn(...)` 改为 facade。
5. 删除 runner 对 `TurnEngineRequest`、`TurnLifecycleRequest`、`TurnLifecycleCompletion` 的直接 import。
6. 增加 focused tests 验证 facade delegation 与 pipeline failure interrupt。

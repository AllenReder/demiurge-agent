## Reference Comparison

### Hermes reference

Hermes keeps delivery routing and display policy in gateway/runtime modules such as `gateway/delivery.py` and `gateway/display_config.py`. Platform adapters still do platform-specific sends, edits, and media handling, but gateway-level output rules are not reimplemented independently by every adapter.

The transferable design is the module shape, not the Hermes implementation. Demiurge should keep `InteractionOutbound`, `InteractionItem`, and the current adapter send methods, but the rule that turns outbound items into delivery steps should live behind one runtime module interface.

### Demiurge current state

This branch has already extracted runtime read-model modules for slash commands, status, tool display, text formatting, event commands, and history display. The remaining shallow outbound delivery cluster is split across:

- `demiurge/ui_gateway/bridge.py`;
- `demiurge/channels/base.py`;
- `demiurge/channels/telegram/bridge.py`.

The three adapters each hand-roll a loop over `outbound.items`:

- text adapters batch adjacent `tool_result` items before `tool_call` or `delivery`;
- Telegram repeats the same text adapter batching while adding platform-specific edit/send logic below it;
- TUI applies the same ordering rule but projects tool results into historical `ToolInteractionRecord.finished(...)` items and batches UI deliveries.

Deletion test: deleting a shared outbound delivery plan module would force these ordering and batching rules back into every adapter.

- Outbound ordering becomes one deeper runtime module interface.
- Locality improves: batching bugs concentrate in one module.

- Adapters keep adapter responsibilities: send/edit/render for their platform.

## Reference Comparison

### Hermes reference

Hermes cron separates trigger bookkeeping from a larger run/delivery execution path:

- `cron/scheduler.py` keeps tick/claim logic around `get_due_jobs(...)`, `advance_next_run(...)`, `mark_job_run(...)`, and a running-job guard.
- `run_job(...)` owns the execution of one claimed cron job: prompt construction, agent invocation, output save, delivery resolution, delivery attempt, and final status recording.
- Delivery routing in Hermes is resolved at fire time, so cron trigger semantics stay separate from platform delivery semantics.

The transferable lesson is not Hermes's job JSON or thread-pool machinery. Demiurge should name the module that executes one claimed schedule fire so the poll/claim module stops owning runner construction, route binding, inbound metadata, control-plane task projection, and completion recording.

### Demiurge current state

This branch now has:

- `SchedulerRuntime` for durable schedule claim, next-fire, single-flight, and run-log read models.
- `ConversationIngressState` / `ConversationTurnController` for human-facing ingress lifecycle across Text, Telegram, and TUI.
- `ChannelCommandRuntime` for external-channel command classification.

The remaining shallow scheduler cluster is `SchedulerService._run_claim(...)`:

- It records `schedule.fire` task projection.
- It validates delivery targets and builds channel routes.
- It constructs a fresh `SessionTurnStepRunner`.
- It builds scheduled `InteractionInbound` metadata.
- It executes selected input/output slots.
- It drains runner background tasks.
- It records durable completion and control-plane completion.

Deletion test: deleting `_run_claim(...)` would force these execution rules to reappear in `SchedulerService` or any future schedule trigger adapter. A named schedule fire module concentrates the execution semantics behind one interface.

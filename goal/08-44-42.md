# 08-44-42 Native Media Delivery Runtime Plan

## Hermes vs Demiurge comparison

- Hermes keeps shared delivery interpretation in `gateway/platforms/base.py`: media/audio routing helpers, reply/thread metadata helpers, and response extraction live above individual platform adapters.
- Hermes Telegram adapter still owns platform I/O: Bot API methods, Telegram voice/audio/document/photo limits, thread retry behavior, and upload fallbacks remain adapter implementation.
- Demiurge already moved text delivery ordering and text route extraction into `demiurge/runtime/outbound_delivery.py`, but `demiurge/channels/telegram/bridge.py` still parses delivery blocks, extracts artifact sources, builds media fallbacks, and decides the text/media sequence.
- The current Telegram delivery module is shallow because callers/tests must know both runtime `InteractionDelivery.blocks` semantics and Telegram upload details to verify one user-visible behavior.

## Breaking redesign target

Deepen outbound delivery by adding a runtime-owned native delivery module behind a small interface:

- Runtime owns `InteractionDelivery` block planning.
- Runtime owns artifact source extraction from `url`, `resolved_path`, and `path`.
- Runtime owns text/media ordering and fallback aggregation.
- Telegram owns only platform-specific sends: photo, voice/audio conversion, video, document, Markdown text formatting, reply anchoring, and Bot API failures.

This preserves the concepts of Agent Core, Agent Slot, and Package capability packages. It changes the host harness/channel implementation shape only.

## Deep module check

- Interface size: the new interface is one runtime call, `NativeDeliveryRuntime.deliver(delivery, target=...)`, plus one adapter callback for media sends and one for text sends.
- Deletion test: deleting the runtime would force `InteractionDelivery.blocks`, artifact source rules, text/media ordering, and fallback aggregation back into every channel adapter that wants native media. The complexity would scatter again.
- Hidden behavior: the module does real behavior, not pass-through. It plans visible delivery items, interprets Agent delivery blocks, resolves artifact sources, preserves ordering, applies first-reply semantics, and aggregates media fallback text.
- Adapter discipline: Telegram keeps only platform differences: Markdown/text send, Bot API upload method choice, reply anchoring, voice conversion, and Telegram failure handling.

## Old interfaces and adapter-local behavior to delete

- Delete Telegram-local `_deliver_media_block(...)`.
- Delete Telegram-local `_artifact_send_source(...)`.
- Stop making `TelegramInteractionBridge._deliver_delivery(...)` parse `delivery.blocks`.
- Stop making Telegram adapter aggregate media fallback text.
- Replace the old adapter-local media path with `_send_native_media(request, target, reply_to=...)`, a narrow adapter method that receives a runtime-planned `NativeMediaRequest`.

## New seam placement

- Runtime seam: `demiurge/runtime/outbound_delivery.py` owns host delivery semantics.
- Adapter seam: `TelegramInteractionBridge` receives already-planned text/media work and performs Telegram-specific I/O.
- Authored Agent Core seam: Agent Core still emits `InteractionDelivery` and artifacts through slots/tools; no authored surface concept changes.
- Package seam: packages can still contribute tools/skills/schedules/lib that produce deliveries/artifacts; package distribution semantics do not change.

## Why this is framework redesign, not ordinary cleanup

- The old shape let a channel adapter know host delivery semantics. That makes bridge/channel code part of the harness lifecycle and blocks a coherent delivery control plane.
- The new shape makes delivery interpretation host-owned, which is the necessary direction for a redesigned Demiurge harness where TUI, channels, dashboard, schedules, and background tasks share one delivery model.
- Hermes is used only as design reference: shared route/media interpretation sits above platform adapters, while platform adapters own platform I/O. No Hermes code is copied.

## New module shape

- Extend `demiurge/runtime/outbound_delivery.py` with native delivery planning data:
  - `NativeMediaRequest`
  - `NativeDeliveryItem`
  - `native_delivery_items(delivery)`
  - `NativeDeliveryRuntime`
- Keep `TextOutboundDeliveryRuntime` as the outer step dispatcher.
- Replace Telegram's local `_deliver_delivery` block loop with `NativeDeliveryRuntime`.
- Replace Telegram's `_deliver_media_block(chat_id, block, reply_to=...)` with a smaller platform adapter method that receives `NativeMediaRequest`.
- Remove Telegram-local artifact source extraction.

## Implementation steps

1. Add runtime planning tests for native delivery item generation:
   - text-only delivery returns text item.
   - mixed text/media blocks preserve order.
   - media request includes kind, source, caption, fallback text, and cloned metadata.
   - invalid media source falls back through existing fallback text behavior.
2. Add runtime execution tests for native delivery:
   - sends text and media in order.
   - only first item receives the original reply anchor.
   - failed media sends one aggregated fallback text.
3. Refactor `demiurge/runtime/outbound_delivery.py` with the new deep module.
4. Refactor Telegram bridge to call the runtime and keep only platform send implementation.
5. Keep existing Telegram behavior and tests passing.

# 09-02-33 Turn Lifecycle Runtime Plan

## Hermes vs Demiurge comparison

- Hermes split the once-per-turn prologue into `agent/turn_context.py` and post-loop finalization into `agent/turn_finalizer.py`. Those modules are named around lifecycle phases, not around generic helpers.
- Hermes still mutates the agent and persistence surfaces during those phases; the useful lesson is locality, not purity.
- Demiurge currently keeps the same kind of host lifecycle in `SessionTurnStepRunner.run_turn()`:
  - turn id creation, `InputEnvelope`, `AgentInput`, `TurnContext`, and `ModuleStateStores`;
  - session-scoped event-log `turn.started` and `message.inbound`;
  - control-plane `agent.turn` submit/start/succeed/fail/cancel;
  - `SessionRuntime.start_turn(...)` and `SessionRuntime.complete_turn(...)`;
  - background completion claim ack through `CompletionInbox`.
- The runner also still owns core loading, route binding, bootstrap, slot pipelines, tool preparation, model loop, output slots, display turns, and history refresh. That makes the runner module shallow at the lifecycle seam: the interface is `run_turn()`, but the implementation forces readers to learn unrelated host lifecycle rules before reaching the model loop.

## Breaking redesign target

Create a runtime-owned `TurnLifecycleRuntime` deep module and delete the runner-local lifecycle helpers.

The module owns host turn lifecycle semantics:

- begin a host turn;
- create the authored-slot context objects needed by the current turn;
- project the turn into the session runtime;
- mirror the turn as an internal `agent.turn` task in the control plane;
- emit inbound/started/completed/failed/cancelled event-log records;
- acknowledge background completion claims after a successful user-visible turn.

This preserves the concepts of Agent Core, Agent Slot, and Package capability packages. It changes the host harness implementation shape only.

## Deep module check

- Interface size: runner should call `begin(...)`, `complete(...)`, and `interrupt(...)`. The caller no longer knows task submission order, runtime-store projection details, or completion-claim shape.
- Deletion test: deleting the module would scatter turn event ordering, task ledger updates, session projection, state-store setup, and completion ack back into runner, scheduler-created runners, child runners, and future dashboard/bridge entry points.
- Hidden behavior: the module performs real behavior, not pass-through. It creates turn identity/context, emits event-log records, writes runtime-store turn rows, drives the control-plane task status, and consumes completion claims.
- Adapter discipline: channel/TUI/Telegram adapters still only provide inbound metadata and delivery routing. Agent Core and Package code still only see slot ctx, tools, skills, state, history, and agents through host-controlled capabilities.

## Old interfaces and runner-local behavior to delete

- Delete `SessionTurnStepRunner._submit_turn_task(...)`.
- Delete `SessionTurnStepRunner._complete_turn_task(...)`.
- Delete `SessionTurnStepRunner._finalize_interrupted_turn(...)`.
- Delete `SessionTurnStepRunner._ack_background_completion_claims(...)`.
- Stop constructing `InputEnvelope`, `AgentInput`, `ModuleStateStores`, and initial `TurnContext` inline in `run_turn()`.
- Stop making `run_turn()` directly know `ActionSpec`, `ActionSource`, or `CompletionInbox`.

## New seam placement

- Runtime seam: `demiurge/runtime/turn_lifecycle.py`.
- Caller seam: `SessionTurnStepRunner.run_turn()` remains the orchestrator for core loading, bootstrap, slot pipelines, tool preparation, model loop, output slots, display history, and final result assembly.
- Lower-level persistence seam: `SessionRuntime` remains the store projection module for sessions/turn rows/messages. It does not own orchestration across event log, task ledger, and completion inbox.
- Control-plane seam: `RuntimeControlPlane` remains the task ledger implementation. It does not know the authored turn context or session event-log concerns.
- Adapter seam: bridges/channels pass `InteractionInbound` metadata into runtime; they do not own host turn lifecycle.
- Authored Agent Core seam: authored slots still receive `TurnContext`, state/history/tool/skill/agent capabilities, and can run Python logic.
- Package seam: packages still distribute tools/skills/schedules/MCP/lib/config and remain outside this host lifecycle change.

## Why this is framework redesign, not ordinary cleanup

- The old shape makes the runner monolith the implicit owner of host lifecycle, which blocks a coherent runtime model for child agents, schedules, bridge-driven turns, dashboard replay, and future cancellation/resume semantics.
- The new shape names host turn lifecycle as a runtime module with a small interface. That is a framework-level seam: it centralizes rules every turn entry point must obey.
- This is the direction needed for a new Demiurge architecture: runner becomes composition of deep runtime modules instead of one method that is simultaneously harness, task ledger, session projector, slot host, and model loop.

## New module shape

- Add `demiurge/runtime/turn_lifecycle.py` with:
  - `TurnLifecycleRequest`
  - `TurnLifecycle`
  - `TurnLifecycleCompletion`
  - `TurnLifecycleRuntime`
- `TurnLifecycleRuntime.begin(request)` returns:
  - `turn_id`
  - `input_envelope`
  - `user_input`
  - `turn`
  - `state_stores`
  - `task_id`
- `TurnLifecycleRuntime.complete(lifecycle, completion)`:
  - emits session event-log `turn.completed`;
  - marks the control-plane task succeeded;
  - completes the session-runtime turn;
  - acknowledges completion claims from inbound metadata.
- `TurnLifecycleRuntime.interrupt(lifecycle, status, error)`:
  - emits session event-log `turn.failed` or `turn.cancelled`;
  - completes the session-runtime turn with the matching status;
  - marks the control-plane task failed or cancelled.

## Implementation steps

1. Add focused tests in `tests/runtime/test_turn_lifecycle.py`:
   - `begin(...)` creates the turn context objects, records `turn.started`, submits/starts `agent.turn`, projects a running turn row, and records `message.inbound`;
   - `complete(...)` emits event-log `turn.completed`, succeeds the control-plane task, completes the runtime turn row, and acks direct and merged completion claims;
   - `interrupt(..., status="failed")` emits `turn.failed`, completes the runtime turn row as failed, and fails the control-plane task with the sanitized error;
   - `interrupt(..., status="cancelled")` emits `turn.cancelled`, completes the runtime turn row as cancelled, and cancels the control-plane task.
   - runner-level cancellation projects a cancelled turn row and cancelled foreground `agent.turn` task.
2. Implement `demiurge/runtime/turn_lifecycle.py`.
3. Refactor `demiurge/runtime/runner.py`:
   - instantiate `TurnLifecycleRuntime`;
   - replace inline begin setup with `lifecycle = self.turn_lifecycle.begin(...)`;
   - pass `lifecycle.turn`, `lifecycle.input_envelope`, and `lifecycle.state_stores` into slot/model flow;
   - replace exception finalization with `self.turn_lifecycle.interrupt(...)`;
   - replace final completion block with `self.turn_lifecycle.complete(...)`;
   - remove deleted helper methods and unused imports.
4. Keep existing behavior in scheduler, child-agent runners, and interaction runtime by constructing the lifecycle inside runner.
5. Do not update public docs unless validation shows a user-visible behavior change; this is internal runtime architecture.

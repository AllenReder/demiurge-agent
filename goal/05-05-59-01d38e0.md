## Reference Comparison

### Hermes reference

Hermes treats `/status` as a read-model assembly point:

- `gateway/slash_commands.py::_handle_status_command(...)` collects session, running-agent, queue, model, context, token, and platform status into one status response.
- `gateway/status.py` owns persisted gateway runtime status helpers used by both runtime status and health surfaces.
- Platform-specific additions, such as Matrix scope details, are appended after the shared status read is assembled.

The transferable lesson is not Hermes's token accounting or platform list. Demiurge should centralize the host-owned status reads and let adapters append UI or platform-specific fields.

### Demiurge current state

This branch now has:

- `demiurge/slash.py` as the command catalog read-model.
- `demiurge/runtime/session_commands.py` as the session list/resume read-model.
- `ConversationIngressState` / `ConversationTurnController` for active turn and queue lifecycle.

The remaining shallow status cluster is:

- `TextChannelBridgeBase._command_status(...)` reads runner core/session/provider/timezone, queue depth, running state, busy mode, and message count.
- `TelegramInteractionBridge._command_status(...)` repeats the same host reads, then appends Telegram access fields.
- `TuiInteractionBridge._status(...)` calls `app.status()` and appends running/busy/queue/tool-display command fields.
- `TuiInteractionBridge._status_payload(...)` repeats status snapshot reads for the frontend event schema.

Deletion test: deleting a shared runtime status read-model module would force core/session/running/queue/message/provider/timezone reads back into TUI, Text, and Telegram adapters.

## This Loop Scope

Extract a runtime status read-model module for command/status surfaces.

This is behaviour-preserving. It does not change TUI status event keys, Telegram access fields, command names, Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, session storage, or provider selection.

## Expected Benefits

- Status command read semantics become one deeper module interface.
- Locality improves: provider/timezone/message-count/running/queue status reads live in one place.
- Leverage improves: Text, Telegram, TUI command output, and future dashboard read surfaces can reuse the same view.
- Tests target the status read-model interface directly while adapters keep platform/UI fields local.

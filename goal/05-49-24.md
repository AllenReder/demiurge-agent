## Reference Comparison

### Hermes reference

Hermes documents observability as a read-only telemetry contract. Its observer hook contract is explicitly backend-neutral, uses stable correlation IDs such as `session_id`, `turn_id`, `tool_call_id`, and keeps tracing/reporting separate from runtime behavior.

The transferable lesson is not to copy Hermes hooks in this loop. Demiurge already emits runtime events. The improvement is to treat user-facing trace/event views as a runtime read-model, not as TUI adapter code.

### Demiurge current state

This branch now has read-model modules for slash commands, sessions, runtime status, tool display, and shared text formatting.

The remaining shallow trace cluster is in `demiurge/ui_gateway/bridge.py`:

- `_trace(...)` resolves `last`, falls back to latest persisted turn, reads event log, builds table rows, and emits text.
- `_events(...)` parses `[type] [limit]`, reads recent events, builds table rows, and emits text.
- `_event_detail(...)` knows how to summarize `actions.requested`, `action.result`, `approval.*`, `message.completed`, and fallback JSON event payloads.

Deletion test: deleting a runtime event command module today would force all trace fallback, event filtering, and detail-summary rules to remain in the TUI adapter or be recopied into any future dashboard/Web UI.

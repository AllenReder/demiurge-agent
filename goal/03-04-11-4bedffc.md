## Reference Comparison

### Hermes reference

Hermes keeps lifecycle intake and delivery concerns in named Modules:

- `gateway/session.py` packages platform origin, session keying, reset/resume policy, and session metadata into explicit records.
- `gateway/stream_dispatch.py` routes typed stream events through adapter render hooks without platform knowledge in the dispatcher.
- `gateway/delivery.py` owns delivery target parsing and fallback routing.
- `tools/process_registry.py` treats background process status, wait/cancel, checkpointing, and completion notification as a first-class runtime lifecycle.

The transferable lesson is that platform adapters should not each own the same lifecycle bookkeeping. A host-owned Module should own the completion event interface, while adapters only decide when to run or queue the resulting inbound turn.

### Demiurge current state

This branch now has named runtime Modules for model/tool loop orchestration, slot phase execution, slot context construction, turn IO, module delivery, and child-agent lifecycle.

The next shallow cluster is background completion intake:

- `demiurge/ui_gateway/bridge.py` claims pending task completion events, converts them into synthetic `InteractionInbound`, merges them into user input, and later relies on runner metadata ack.
- `demiurge/channels/base.py` repeats the same claim/inbound/merge helpers for generic text channels.
- `demiurge/channels/telegram/bridge.py` repeats the same claim/inbound/merge helpers again for Telegram.
- `demiurge/runtime/runner.py` owns `_ack_background_completion_claims(...)`, which parses the same metadata shape that the adapters synthesize.

Deletion test: deleting one adapter's helper block would force the same completion metadata, claim ownership, and merge rules to be reimplemented locally. This is a shallow module repeated across real adapters. Two adapters justify the seam; three make it overdue.

## This Loop Scope

Extract a `CompletionInbox` runtime Module that owns background task completion inbound construction, metadata merge, durable claim creation, and ack extraction.

This is behavior-preserving. It changes internal Module placement and adapter call sites only; it does not change Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, task tool semantics, queue/interrupt behavior, channel command behavior, TUI rendering, or Telegram delivery behavior.

## Expected Benefits

- Background completion intake becomes one deep Module Interface instead of three copied helper clusters.
- Locality improves: claim ownership, synthetic inbound metadata, and ack rules live together.
- Leverage improves: TUI, generic text channels, Telegram, and runner turn commit all use the same Interface.
- The later `IngressTurnController` can depend on completion semantics without inheriting adapter-specific helper code.
- Tests can exercise completion merge and ack behavior without constructing a TUI or Telegram adapter.

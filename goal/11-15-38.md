# 11-15-38 Session Routing Runtime Plan

## Hermes vs Demiurge comparison

- Hermes gateway keeps route identity as a first-class session concern: platform source data becomes a deterministic `session_key`, and `SessionStore` owns key construction, session lookup, reset/switch, active entry metadata, and transcript routing.
- Hermes adapters provide source facts such as platform, chat, user, thread, and profile. They do not decide whether an inbound message should resume an old conversation, bind the current one, or create a fresh session.
- Current Demiurge has the same conceptual need, but the logic is split:
  - `SessionTurnStepRunner._interaction_metadata(...)` normalizes inbound route metadata and timezone.
  - `SessionTurnStepRunner._resolve_session_for_interaction(...)` decides whether to bind current session, switch to an existing one, or create a new one.
  - `SessionTurnStepRunner.start_new_session(...)` owns explicit new-session creation and conversation rebind semantics.
  - `SessionTurnStepRunner.resume_session(...)` owns manual resume.
  - `session_commands.py` then wraps part of that for UI/channel commands.
- This keeps a host lifecycle rule inside the runner facade. It also makes route/session behavior look like an adapter concern, even though adapter differences should stop at route facts.

## Breaking redesign target

Introduce `SessionRoutingRuntime` as the deep module for host-owned session routing:

- The module owns interaction metadata normalization.
- The module owns initial current-session ensure.
- The module owns explicit new session, manual resume, and inbound route resolution.
- The module decides whether route metadata updates the current session, switches to an existing session, or creates a fresh session.
- `SessionRuntime` remains the storage and event projection module for sessions, turns, messages, compaction, and delivery.
- `SessionTurnStepRunner` becomes a composition facade: it loads core binding facts, delegates route decisions, and only keeps active-session rebind mechanics.

Agent Core, Agent Slot, and Package concepts remain unchanged.

## Deep module check

- Interface size: runner should only call `session_routes.ensure_current(binding)`, `metadata_for(interaction)`, `resolve_for_interaction(binding, metadata)`, `start_new(binding, ...)`, and `resume(session_id)`.
- Deletion test: deleting the module would scatter interaction metadata normalization, route binding, session switching, event emission, and new-session binding conflict behavior back across runner, turn pipeline host, and UI command helpers.
- Hidden behavior: the module has real behavior. It hides route-to-session policy, empty conversation binding rules, rebind ordering, metadata filtering, and session-created/resumed event emission.
- Adapter discipline: UI gateway, Telegram/channel adapters, and future dashboards continue to provide route facts only; they do not learn session binding policy.

## Old runner-local behavior to delete or move

- Move `_interaction_metadata(...)` out of runner.
- Move `_resolve_session_for_interaction(...)` out of runner.
- Move `start_new_session(...)` session creation internals out of runner.
- Move `resume_session(...)` session lookup/event behavior out of runner.
- Move `_ensure_current_session(...)` session ensure behavior out of runner.
- Replace `_switch_session(...)` with a private activation hook that only updates the active session id, event log, and cached history.

## New seam placement

- New module: `demiurge/runtime/session_routing.py`
  - `SessionCoreBinding`: core id, revision, provider, model, workspace facts needed for session records.
  - `SessionRoutingRuntime`: route/session policy and event emission.
- Runner composition:
  - runner creates `self.session_routes = SessionRoutingRuntime(...)`.
  - runner maps a `LoadedCore` to `SessionCoreBinding`.
  - runner public methods stay as compatibility facades for current callers, but their implementations delegate.
- Turn pipeline host:
  - `interaction_metadata(...)` delegates to `runner.session_routes.metadata_for(...)`.
  - `resolve_session_for_interaction(...)` delegates to `runner.session_routes.resolve_for_interaction(...)`.

## Why this is framework redesign, not ordinary cleanup

- This moves session route ownership out of the harness runner and into a module whose interface can become the future route/thread control plane.
- It aligns with Hermes' stronger separation: adapters describe source identity, a session module owns routing policy, and the conversation loop does not carry platform/session-key condition ladders.
- It prepares later dashboard/TUI/channel redesign because operator UIs and messaging channels can share one route/session module instead of reimplementing resume/new/bind behavior.

## Implementation steps

1. Add `demiurge/runtime/session_routing.py` with `SessionCoreBinding` and `SessionRoutingRuntime`.
2. Wire `SessionTurnStepRunner` to construct `SessionRoutingRuntime`.
3. Replace runner session methods with delegating facades.
4. Update `RunnerTurnPipelineHost` to use the new route module.
5. Add focused tests for metadata normalization, inbound route switching, route binding to current empty sessions, explicit rebind, and manual resume.
6. Run targeted session/turn/runtime tests.

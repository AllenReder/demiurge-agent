# 11-45-32 Conversation Lifecycle Runtime Plan

## Global architecture review

The last several rounds moved turn pipeline, slot execution, prompt context,
session compaction, session routing, and slot effect handling out of
`SessionTurnStepRunner`. That was useful, but the remaining high-friction shape
is now outside the runner:

- `demiurge/channels/base.py` and `demiurge/channels/telegram/bridge.py` both
  duplicate conversation-key state lookup, busy-turn queueing, active task
  lifecycle, completion subscription, completion enqueue, and drain-next logic.
- `demiurge/runtime/ingress.py`, `conversation_states.py`, and
  `completion_delivery.py` already contain most of the real behavior, but they
  are still called piecemeal from adapters.
- `demiurge/ui_gateway/bridge.py` has a related but operator-console-specific
  path with prompt/approval/status UI concerns; this round will not force it
  into the channel shape.

## Hermes vs Demiurge comparison

- Hermes gateway keeps the per-conversation lifecycle in one gateway runtime:
  session keys, running-agent slots, busy input queueing, pending follow-up
  drain, and completion/resume behavior are owned by gateway lifecycle code.
- Hermes platform adapters carry platform-specific transport facts. They do
  not each own a separate implementation of "when a turn is running, queue or
  interrupt, then drain the next inbound and merge background completions".
- Current Demiurge has the right primitives, but the module is shallow: the
  adapter still knows how to compose them and must repeat that composition.

## Breaking redesign target

Introduce `ConversationLifecycleRuntime`:

- Own state lookup and task-worker completion subscription through
  `ConversationStateStore`.
- Own busy input queue/interrupt decisions through `ConversationTurnController`.
- Own pending background completion merge/enqueue through
  `CompletionDeliveryRuntime`.
- Own task start/finish/drain-next wrapping for one conversation state.
- Leave transport behavior in adapters:
  - command parsing and platform command availability;
  - sending queue/interrupt notices;
  - sending typing indicators;
  - delivery rendering;
  - Telegram approval UI and callback routing.

This keeps Agent Core, Agent Slot, and Package concepts unchanged. It moves
host lifecycle rules out of channel adapters.

## Deep module check

- Interface size: adapters should call `state_for_key(...)`,
  `remember_route(...)`, `merge_pending(...)`, and `accept_inbound(...)`; slash
  commands that already operate on a selected state call `run_state_turn(...)`
  when they manually drain queued input. The runtime internally owns
  finish/drain and task completion enqueue.
- Deletion test: deleting this module would re-scatter state store ownership,
  task worker subscription, completion event claiming, active task finish,
  drain-next, and busy input policy across every channel adapter.
- Hidden behavior: the module hides real lifecycle behavior, not a pass-through:
  it wraps the active task, guarantees finish/drain on completion, and claims
  pending completion events exactly once.
- Adapter discipline: Text and Telegram adapters retain platform differences;
  host lifecycle leaves adapters.

## Old interfaces and adapter-local behavior to delete

- Delete `ConversationStateStore` construction from `TextChannelBridgeBase` and
  `TelegramInteractionBridge`.
- Delete adapter-local `_start_turn(...)`, `_on_task_completion(...)`,
  `_enqueue_task_completion(...)`, and `_completion_delivery_runtime(...)`.
- Remove duplicated finish/drain logic from adapter `_run_inbound(...)`.
- Keep command-level `ConversationTurnController` usage in
  `demiurge/channels/commands.py` for now, because slash commands operate on
  an already selected state and expose command-specific behavior.

## New seam placement

- New module: `demiurge/runtime/conversation_lifecycle.py`
  - `ConversationLifecycleRuntime`
  - `ConversationLifecycleConfig`
  - callback types for `run_turn`, `notify_busy`, and optional
    `before_completion_enqueue`
- Text channel bridge:
  - constructs `ConversationLifecycleRuntime` with `channel_name`, owner ids,
    state factory, `_run_inbound`, and busy notifier.
- Telegram bridge:
  - constructs `ConversationLifecycleRuntime` with Telegram owner ids, state
    factory, `_run_inbound`, and Telegram busy notifier.

## Why this is framework redesign, not ordinary cleanup

- It changes the seam between runtime and adapters. Channel adapters stop
  composing host lifecycle primitives and become transport adapters.
- It moves Demiurge closer to a Hermes-style gateway/runtime split: platform
  adapters normalize and deliver; runtime owns conversation lifecycle.
- It prepares a later operator-console/TUI pass because there will be one
  runtime module to adapt, rather than two channel-specific patterns plus one
  TUI-specific pattern.

## Implementation steps

1. Add `demiurge/runtime/conversation_lifecycle.py`.
2. Add focused tests for busy queue/start/drain and task completion enqueue.
3. Migrate `TextChannelBridgeBase` to the new module.
4. Migrate `TelegramInteractionBridge` to the new module.
5. Run targeted runtime/channel tests.

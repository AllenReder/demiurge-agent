# Turn Projection Task Column Removal Plan

## Hermes vs Demiurge Comparison

Hermes frames ordinary LLM turns as session lifecycle records. Turn-level hooks
carry `session_id` and `turn_id`; task identity appears only when execution is
inside an actual delegated or managed task context.

Demiurge now follows the same framework shape in behavior: foreground Agent
Core turns do not enter the task ledger, `task_list` ignores them, and
`task_status` / `task_control` reject turn ids. The remaining mismatch is the
session turn projection itself: `SessionRuntime.start_turn()` still accepts a
`task_id` argument, `turn.started` stores it, and the `turns` table exposes a
nullable `task_id` column that tests only assert is `None`.

## Framework Shape Decision

The `turns` projection should describe session turn state only:

- `turn.started` carries `session_id`, status, and input/result references.
- `SessionRuntime.start_turn()` no longer accepts `task_id`.
- `turns` stores no task ownership column.
- Real detached work continues to use the `tasks`, `task_logs`, and
  `runtime_work_items` projections.

This concentrates session-turn locality in the session module and keeps task
lineage local to the control-plane module.

## Delete Or Merge

- Delete `task_id` from `SessionRuntime.start_turn()`.
- Delete `task_id` from `turn.started` payload projection and `turns` schema.
- Update `TurnLifecycle` to call `start_turn()` without a task argument.
- Update regression tests to assert foreground turn ids are absent from the
  task ledger without checking an always-null turn column.

## Behavior To Keep

- Foreground turns still produce `turn.started` runtime events.
- Session snapshots still expose turn status, input references, result
  references, and completion timestamps.
- `task_status` and `task_control` still reject foreground turn ids.
- Background task records, logs, completion events, and task lineage are
  unchanged.

## Why This Is A Demiurge Framework Step

This removes a shallow compatibility-shaped interface from the session module.
The module becomes deeper because callers no longer need to understand a
nullable task field that is deliberately unusable for foreground turns.

## Implementation Steps

1. Remove the `task_id` parameter and payload field from
   `SessionRuntime.start_turn()`.
2. Update `TurnLifecycle.begin()` to call `start_turn()` with session and turn
   identity only.
3. Remove `task_id` from the `turns` projection insert list and schema.
4. Update foreground turn tests to prove the task ledger rejects turn ids and
   that turn rows retain their session-local fields.
5. Run targeted stale-reference search, session/turn lifecycle tests,
   compileall for touched files, and scoped diff checks.
6. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

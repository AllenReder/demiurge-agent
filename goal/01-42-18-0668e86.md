## Reference Comparison

### Hermes reference

Hermes is useful here as a lifecycle naming reference:

- `agent/turn_context.py` moves once-per-turn setup into a named Module that produces a compact context product.
- `agent/turn_finalizer.py` moves post-loop cleanup into a named Module that consumes explicit locals.
- `agent/conversation_loop.py` keeps the model/tool loop more navigable because those lifecycle products are named.

The transferable lesson is not Hermes code shape. Demiurge has Agent Slots, so the deeper Module should be the authored slot context lifecycle: build the SDK `ctx` object for a single slot invocation from a compact request and host adapter.

### Demiurge current state

The branch already has:

- `TurnEngine` behind `TurnEngineHost`.
- `SlotPipelineRuntime` for input/output phase scheduling and result assembly.
- `InputSlotRunRequest` and `OutputSlotRunRequest` so the slot pipeline host seam no longer exposes long parameter lists.

The remaining friction is `SessionTurnStepRunner` still owns authored-surface `ctx` construction:

- `InputContext(...)` and `OutputContext(...)` are assembled inline in runner.
- SDK clients for `ctx.input`, `ctx.output`, `ctx.history`, `ctx.state`, `ctx.tools`, `ctx.agents`, `ctx.skills`, `ctx.result`, and phase-local delivery are runner-local classes.
- This means any authored `ctx.*` change still touches the harness god Module.

Deletion test: deleting the runner's input/output slot invocation blocks would force the same context-construction logic to reappear wherever slots are invoked. That is a shallow Module.

## This Loop Scope

Implement the next deepening candidate:

> Extract a `SlotContextRuntime` Module that owns input/output Agent Slot SDK context construction and phase-local SDK clients.

This loop is intentionally behavior-preserving. It changes internal Module placement, not Agent Core layout, slot manifests, package recipes, or channel behavior.

## Planned Code Changes

- Add `demiurge/runtime/slot_context.py`.
- Move the following runner-local SDK/client classes into that Module:
  - `ModuleStateStores`
  - `ScopedModuleStateClient`
  - `ModuleStateClient`
  - `ModuleToolClient`
  - `ModuleAgentsClient`
  - `ModuleInputClient`
  - `ModuleOutputClient`
  - `ModuleHistoryClient`
  - `ModuleSkillClient`
  - `ModuleIOClient`
  - `ModuleResultClient`
- Keep `ModuleBootstrapClient` in runner for this loop; bootstrap is still outside the input/output slot pipeline.
- Add a `SlotContextHost` Protocol and `RunnerSlotContextHost` Adapter so clients can call host-owned effects without importing the runner Module.
- Add `SlotContextRuntime.build_input_context(...)` and `build_output_context(...)`, returning the SDK context and its phase-local IO client.
- Instantiate `self.slot_context = SlotContextRuntime(RunnerSlotContextHost(self))` in `SessionTurnStepRunner`.
- Replace inline `InputContext(...)` / `OutputContext(...)` construction in `_run_input_slot(...)` and `_run_output_slot(...)` with calls into `self.slot_context`.
- Keep actual slot invocation, skill activation bookkeeping, soft/hard failure policy, background flushing, delivery scheduling, tool execution, and child-agent run/spawn behavior host-owned.
- Add focused tests in `tests/runtime/test_slot_context.py` for context shape and background restrictions.

## Out Of Scope

- Moving bootstrap execution or `ModuleBootstrapClient`.
- Moving full `SlotRuntime.invoke(...)` into `SlotContextRuntime`.
- Extracting child-agent runtime.
- Extracting module delivery runtime.
- Changing `ctx.history` semantics.
- Changing `ctx.state.core` / `ctx.state.session`.
- Changing Agent Core file layout, slot YAML, pipelines YAML, or package recipes.
- Copying Hermes implementation code.

## Expected Benefits

- `SessionTurnStepRunner` loses authored SDK client construction responsibility.
- `SlotContextRuntime` becomes the direct test surface for input/output `ctx` shape.
- Locality improves: `ctx.*` rules live in one Module.
- The next child-agent and module-delivery deepening candidates get a cleaner host seam.

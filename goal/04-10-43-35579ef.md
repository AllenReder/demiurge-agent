## Reference Comparison

### Hermes reference

Hermes keeps active-session lifecycle rules in the shared gateway adapter module:

- `gateway/platforms/base.py` owns `_active_sessions`, `_pending_messages`, busy text mode, interruption, cancellation, and pending-drain handoff.
- The same module also protects control commands that must bypass an active session and then drain queued follow-up messages.
- Hermes tests such as `tests/gateway/test_pending_drain_race.py` and `tests/gateway/test_command_bypass_active_session.py` treat those rules as gateway invariants rather than Telegram-only behaviour.

The transferable lesson is not Hermes's exact `MessageEvent` or guard map. Demiurge should keep one host-owned ingress lifecycle module and make every human-facing adapter cross that seam.

### Demiurge current state

This branch now has:

- `CompletionInbox` for background completion claim and ack.
- `InboundQueueRuntime` for inbound queue selection and completion merge.
- `ConversationIngressState` for route memory and completion-route construction.
- `ConversationTurnController` for active-task, queue-drain, cancel, and completion-enqueue lifecycle rules.
- `ChannelCommandRuntime` for external-channel slash command classification and dispatch.

`TextChannelBridgeBase` and `TelegramInteractionBridge` now delegate active-task and queue/completion lifecycle to `ConversationTurnController`.

`TuiInteractionBridge` still carries its own copy:

- `_running_task`;
- `_queued_inputs`;
- `submit(...)` busy queue vs interrupt rules;
- `_merge_pending_completions_into(...)`;
- `_drain_next_queued_input(...)`;
- `_enqueue_task_completion(...)`;
- direct active-task cancellation in `interrupt_current_turn(...)` and `shutdown(...)`.

Deletion test: deleting TUI's local lifecycle code would currently remove behaviour that must be rebuilt there, even though the same behaviour already exists in the runtime ingress module for Text and Telegram. Three adapters justify the seam strongly.

## This Loop Scope

Make `TuiInteractionBridge` an adapter over `ConversationIngressState` and `ConversationTurnController`.

This is behaviour-preserving. It changes internal module placement and adapter call sites only; it does not change Agent Core layout, Agent Slot manifests, Package recipes, authored SDK methods, TUI command names, prompt/approval protocol, visible command output, history snapshot shape, busy mode meanings, delivery rendering, or task completion schema.

## Planned Code Changes

- Update `demiurge/ui_gateway/bridge.py`:
  - import and instantiate `ConversationIngressState` and `ConversationTurnController`;
  - keep one TUI ingress state bound to `InteractionRuntime(app.runner)` and the existing `SessionRouteBinding`;
  - make `running` read the controller instead of `_running_task`;
  - make `submit(...)` delegate busy queue/interrupt decisions to the controller while keeping TUI notices local;
  - make idle submit merge stored completions through the ingress state's queue and start through the controller;
  - make `_run_inbound(...)` finish through the controller and drain through the controller;
  - make `_enqueue_task_completion(...)` claim through the ingress state and enqueue/start through the controller;
  - make `interrupt_current_turn(...)` cancel through the controller while keeping TUI notice text local;
  - update `_new(...)`, `_resume_session(...)`, and `/busy` to keep the ingress state's runtime, conversation key, and busy mode current.
- Preserve private `_queued_inputs` as an alias to the ingress queue while existing focused TUI tests still assert queue state.
- Remove TUI-local completion route construction that duplicates `ConversationIngressState`.
- Add or update focused tests only where the TUI adapter's existing tests do not already prove the controller-backed behaviour.

## Expected Benefits

- The conversation ingress lifecycle becomes one deeper module interface across Text, Telegram, and TUI.
- Locality improves: active-task state, busy-mode decisions, queue drain, and completion enqueue rules live in one module.
- Leverage improves: future Web UI or dashboard adapters can reuse the same controller instead of copying TUI logic.
- Tests target the shared lifecycle interface plus adapter-level rendering, matching the codebase-design principle that the interface is the test surface.

# 10-42-21 Slot Execution Runtime Plan

## Hermes vs Demiurge comparison

- Hermes 没有 Demiurge 的 Agent Slot 概念，但它把可插拔执行点的 host 规则放在 runtime/helper module 中：platform/gateway 不知道具体工具、context engine、finalizer 的内部顺序，agent facade 也不应该散落每个执行点的 failure policy 和 side-effect commit。
- Demiurge 的 Agent Slot 是必须保留的核心概念，但当前 module seam 是错的：
  - `SlotPipelineRuntime` 只决定 serial/parallel phase 顺序；
  - `SlotContextRuntime` 只构造 SDK ctx；
  - `SlotEffectRuntime` 只 commit delivery/effect；
  - 真正的“单个 authored slot 如何运行”仍在 `SessionTurnStepRunner`。
- 这导致 runner 还知道太多 authored contract：slot return 值忽略规则、skill activation capability、slot-end delivery scheduling、soft/hard failure policy、input/output ctx build 差异、output 后 history refresh。

## Breaking redesign target

新增 `SlotExecutionRuntime`，让 Agent Slot 的单次执行 lifecycle 成为深 module：

- `SlotPipelineRuntime` 继续拥有 phase ordering。
- `SlotExecutionRuntime` 拥有单个 input/output slot 的执行 contract。
- `SlotRuntime` 继续只负责调用 authored Python callable。
- `SlotContextRuntime` 继续只负责构造作者可见 ctx。
- `SlotEffectRuntime` 继续只负责 delivery/effect commit 和 dispatch。
- `SessionTurnStepRunner` 不再实现 `_run_input_slot`、`_run_output_slot`、`_require_skill_activation`、`_call_slot`。

Agent Core、Agent Slot、Package 的概念不变。本轮改变的是 host 对 Agent Slot 的内部 ownership，不改变作者可见能力。

## Deep module check

- Interface size: runner 只调用 `slot_execution.run_input(request)` 和 `slot_execution.run_output(request)`。
- Deletion test: 删除该 module 会把 ctx build、slot callable invocation、return ignored event、skill activation、delivery scheduling、failure policy、history refresh 重新散回 runner。
- Hidden behavior: module 承担真实 slot contract，不是 pass-through。它把 `SlotPipelineRuntime` 的 phase request 转成一次完整的 authored slot lifecycle。
- Adapter discipline: channel/TUI/dashboard 不知道 slot lifecycle；turn pipeline 也只通过 slot pipeline 间接触达 slot execution。Agent Slot 的 host enforcement 留在 runtime，不进入 adapter。

## Old interfaces and runner-local behavior to delete

- 删除 runner `_run_input_slot(...)`。
- 删除 runner `_run_output_slot(...)`。
- 删除 runner `_require_skill_activation(...)`。
- 删除 runner `_call_slot(...)`。
- 将 `run_input_pipeline_slot(...)` 和 `run_output_pipeline_slot(...)` 改为委托到 `SlotExecutionRuntime`。

## New seam placement

- New module: `demiurge/runtime/slot_execution.py`
  - `SlotExecutionRuntime.run_input(...)`
  - `SlotExecutionRuntime.run_output(...)`
  - private `_call_slot(...)`
  - private `_require_skill_activation(...)`
- Runner composition:
  - 在 `slot_context` 和 `slot_effects` 创建后构造 `self.slot_execution`。
  - `RunnerSlotPipelineHost` 仍调用 runner facade；runner facade 再委托到 slot execution。后续可进一步让 pipeline host 直接拥有 slot execution。

## Why this is framework redesign, not ordinary cleanup

- Agent Slot 是 Demiurge 的核心概念之一；本轮把 slot 的 host contract 从 legacy runner 移入专门 runtime module。
- 这让新的 harness 形状更清楚：turn pipeline -> slot pipeline -> slot execution -> slot callable/context/effects。
- 这一步为后续破坏式重写 slot result/effect model、capability enforcement、parallel isolation、child-agent ctx interface 提供单一 seam。

## Implementation steps

1. 新增 `demiurge/runtime/slot_execution.py`，迁移 input/output slot 单次执行逻辑。
2. 在 runner 初始化 `self.slot_execution`。
3. 将 `run_input_pipeline_slot(...)`、`run_output_pipeline_slot(...)` 改为委托。
4. 删除 runner 中已迁移的 private methods。
5. 增加 focused tests 覆盖：
   - input slot activation 通过 capability 后追加 skill contribution；
   - soft failure 返回已产生 items 并标记 pending failed；
   - output slot 成功后刷新 history。

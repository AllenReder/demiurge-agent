# 09-26-03 Slot Effect Runtime Plan

## Hermes vs Demiurge comparison

- Hermes split agent-to-gateway presentation into typed stream events in `gateway/stream_events.py` and adapter-driven dispatch in `gateway/stream_dispatch.py`.
- The useful lesson is not the exact event names. The useful seam is: authored/agent logic emits structured facts, runtime dispatch owns routing state, and adapters decide platform presentation.
- Demiurge's Agent Slot concept is stronger than Hermes' plugin callbacks, but the current implementation still routes slot effects through a shallow chain:
  - `ctx.output.send_text(...)` / `ctx.input.send_text(...)` live on `ModuleIOClient` in `slot_context.py`;
  - `SlotContextRuntime` passes runner callback lambdas into that client;
  - runner callback `commit_module_delivery_request(...)` calls `TurnIO.send_module_output(...)`;
  - `TurnIO` calls back into runner `_apply_delivery_request(...)`;
  - `ModuleDeliveryRuntime` commits history/artifacts/outbox;
  - runner then owns route metadata, dispatch status, background delivery task creation, pending-item flush, and failure marking.
- This makes `SessionTurnStepRunner` a hidden owner of authored slot effects even after turn lifecycle was extracted.

## Breaking redesign target

Create a runtime-owned slot effect seam:

- `SlotEffectRuntime` owns authored slot delivery-effect commit.
- `InteractionDispatchRuntime` owns interaction item scheduling, immediate dispatch, pending flush, dispatch metadata, and pending-item failure marking.
- `TurnIO` stops knowing how to send module output.
- `SessionTurnStepRunner` stops exposing slot-effect callback methods.

Agent Core, Agent Slot, and Package concepts are unchanged. This only changes host runtime ownership.

## Deep module check

- Interface size: slot context should ask one runtime module for a sender and the runner should call a small dispatch interface. Slot authors still see `ctx.output.send_text(...)`.
- Deletion test: deleting the new seam would scatter delivery request commit, route metadata merging, dispatch status, background dispatch tasks, and slot-failure marking back into runner, `TurnIO`, and slot context callbacks.
- Hidden behavior: the module does real behavior. It validates and commits `DeliveryRequest` through module delivery, wraps visible deliveries into interaction items, schedules them, flushes background items, and records failed pending items on slot failure.
- Adapter discipline: platform adapters still only deliver `InteractionOutbound`. They do not own host effect semantics. Authored Agent Core and Packages still only emit effects through host-controlled `ctx`.

## Old interfaces and runner-local behavior to delete

- Delete `TurnIO.send_module_output(...)`.
- Delete runner `commit_module_delivery_request(...)`.
- Delete runner `_commit_module_delivery_request(...)`.
- Delete runner `_apply_delivery_request(...)`.
- Delete runner `_apply_deliver_effect(...)`.
- Stop exposing `commit_module_delivery_request` and `schedule_interaction_item` from `SlotContextHost`.
- Move runner scheduling helpers behind `InteractionDispatchRuntime`:
  - `schedule_interaction_item`;
  - `_schedule_interaction_item`;
  - `_dispatch_interaction_item_now`;
  - `_flush_pending_background_items`;
  - `_schedule_slot_end_delivery_items`;
  - `_mark_slot_end_delivery_failed`;
  - interaction outbound metadata helpers.

## New seam placement

- Slot effect seam: `demiurge/runtime/slot_effects.py`.
- Interaction dispatch seam: `demiurge/runtime/interaction_dispatch.py`.
- Module delivery seam: `demiurge/runtime/module_delivery.py` remains the persistence/artifact/outbox implementation for `DeliveryRequest`.
- Turn IO seam: `demiurge/runtime/io.py` remains model-loop IO for user, assistant step, and tool call/result messages.
- Slot context seam: `demiurge/runtime/slot_context.py` remains the builder for authored `ctx`; it delegates effect commit to `SlotEffectRuntime`.
- Adapter seam: bridges/channels keep platform delivery only.
- Authored Agent Core and Package seams are unchanged.

## Why this is framework redesign, not ordinary cleanup

- The old shape makes the runner a hidden transaction coordinator for authored slot effects. That blocks a coherent Agent Slot runtime because delivery/history/dispatch rules are not owned by the slot runtime.
- The new shape makes authored slot effects a named runtime module. It is a step toward a redesigned Demiurge harness where slots produce host-mediated effects and the runner composes deep runtime modules.
- Hermes is used only as design reference: typed facts cross a runtime seam, presentation stays in adapters, and delivery routing rules do not live in the agent loop.

## New module shape

- Add `demiurge/runtime/interaction_dispatch.py`:
  - `InteractionDispatchRuntime.schedule(...)`;
  - `InteractionDispatchRuntime.dispatch_now(...)`;
  - `InteractionDispatchRuntime.flush_pending(...)`;
  - `InteractionDispatchRuntime.mark_pending_failed(...)`.
- Add `demiurge/runtime/slot_effects.py`:
  - `SlotEffectRuntime.module_io_client(...)`;
  - `SlotEffectRuntime.result_client(...)`;
  - `SlotEffectRuntime.commit_delivery_request(...)`;
  - `SlotEffectRuntime.apply_deliver_effect(...)`;
  - `SlotEffectRuntime.schedule_interaction_item(...)`;
  - `SlotEffectRuntime.flush_background_items(...)`;
  - `SlotEffectRuntime.mark_pending_failed(...)`.
- Wire `SessionTurnStepRunner` to instantiate both modules and pass `SlotEffectRuntime` into `SlotContextRuntime`.
- Keep `ModuleIOClient` / `ModuleResultClient` authored SDK surface unchanged in this loop; moving those class definitions can be a later layout cleanup after the ownership seam is real.

## Implementation steps

1. Add focused tests for interaction dispatch:
   - scheduled item starts a background dispatch task and marks status scheduled;
   - no channel marks item unrouted;
   - flush dispatches pending items synchronously;
   - mark pending failed annotates delivery metadata.
2. Add focused tests for slot effects:
   - committing a text delivery request returns a delivery interaction item and writes history/outbox through `ModuleDeliveryRuntime`;
   - applying a legacy deliver effect uses the same commit path;
   - `module_io_client(...).send_text(...)` commits and schedules through the runtime seam.
3. Refactor `SlotContextRuntime` to ask host `slot_effects` for IO/result clients instead of constructing callback lambdas to the runner.
4. Refactor `TurnIO` to delete module-output delivery.
5. Refactor `SessionTurnStepRunner` to use `InteractionDispatchRuntime` and `SlotEffectRuntime`, then delete old slot-effect helpers.
6. Keep Agent Slot authored behavior and existing tests passing.

# Conversation Lifecycle Submission Seam Plan

## Hermes vs Demiurge Comparison

Hermes keeps platform adapters focused on platform facts: chat identity,
thread metadata, slash-command surfaces, and message delivery. Session turn
lifecycle, queued input policy, task-completion intake, and run scheduling are
owned by the gateway/harness layer instead of being reassembled by each
adapter.

Demiurge has a `ConversationLifecycleRuntime` module, but its interface is
still shallow. TUI and text channel adapters have to know the sequencing:
remember the route, check whether a turn is running, handle busy/interrupt
policy, optionally wait for the interrupted task, merge pending background
completions, and finally accept/start the inbound turn. Deleting the lifecycle
module would move that complexity back into adapters, which shows the module
should be deeper.

## Framework Shape Decision

Conversation submission should cross one runtime seam:

- Adapters construct `InteractionInbound` and call one runtime method.
- `ConversationLifecycleRuntime` owns route memory, busy handling, interrupt
  cancellation waiting, pending completion merge, and turn start.
- Adapters remain responsible for platform-specific command parsing, prompt
  resolution, status payloads, and rendered notices.
- Background task completion ownership stays in runtime/task modules; channel
  adapters only render platform messages or UI status.

This keeps platform adapter differences at the adapter seam and keeps host
lifecycle rules local to a deep runtime module.

## Delete Or Merge

- Add a `ConversationSubmitResult` return type and a
  `ConversationLifecycleRuntime.submit_inbound(...)` method.
- Merge the duplicated manual sequence from `TuiInteractionBridge.submit()` and
  `TextChannelBridgeBase.handle_inbound()` into that method.
- Delete direct adapter calls to `merge_pending(...)` + `accept_inbound(...)`
  for normal user submissions.
- Keep lower-level `handle_busy`, `merge_pending`, and `accept_inbound` only as
  internal/testable primitives for lifecycle-specific tests and commands that
  need them.

## Behavior To Keep

- TUI busy mode still queues or interrupts exactly as before.
- TUI interrupt mode still waits for the cancelled active turn before returning
  the queued result and emits the same notices.
- Pending background completions still merge into the next user turn before
  that turn runs.
- Background completion inbound turns still do not echo as user messages in
  the TUI transcript.
- Text/Telegram channels keep their prompt, approval, delivery, and command
  behavior.

## Why This Is A Demiurge Framework Step

This changes framework shape rather than just moving code. The runtime
conversation lifecycle module becomes the interface for host turn submission,
while adapters become thinner adapters. That is closer to the intended
Demiurge architecture: host lifecycle in runtime modules, platform differences
in adapters, authored Agent Core and Package concepts unchanged.

## Implementation Steps

1. Add `ConversationSubmitResult` to `demiurge.runtime.conversation_lifecycle`.
2. Implement `submit_inbound(...)` so it remembers route, handles busy turns,
   optionally waits for interrupted active turns, merges pending completions
   when idle, and starts the turn.
3. Refactor `TuiInteractionBridge.submit()` to call `submit_inbound(...)` and
   derive its public payload from the result.
4. Refactor `TextChannelBridgeBase.handle_inbound()` to call
   `submit_inbound(...)` after command and prompt resolution.
5. Add lifecycle tests for idle submission, busy interrupt waiting, and pending
   completion merge through the new interface; keep adapter tests as behavior
   regression checks.
6. Validate conversation lifecycle, ingress, UI gateway, and Telegram channel
   paths; compile touched runtime/channel files; run scoped diff checks.
7. Commit implementation, then rename this plan to include the implementation
   commit hash and commit that rename separately.

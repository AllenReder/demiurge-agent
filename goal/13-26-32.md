# Foreground Turn Task Ledger Split Plan

## Hermes vs Demiurge Comparison

Hermes keeps foreground conversation turns on the session/thread lifecycle seam:
the turn prologue builds per-turn context, the finalizer persists session
history and cleanup results, and platform gateway code routes messages by
session/thread metadata. Background process completion is handled by separate
watchers/registries that send status notifications back to the relevant
thread. That shape distinguishes "the user-facing conversation is running" from
"a detached piece of work can be listed, waited on, cancelled, and reported
later".

Current Demiurge has the wrong module shape at this seam. `RuntimeTaskWorker`
already treats tasks as detached background work (`agent.spawn`,
`terminal.exec`, `evolver.run`), but `TurnLifecycleRuntime` still bypasses that
deep module and writes every foreground turn into `RuntimeControlPlane` as an
`agent.turn` task. The model-facing task tools hide those rows, but the host
still records ordinary turns as tasks. That leaves the task ledger conceptually
shallow: it must know about session turns that it should not control.

## Framework Shape Decision

The new seam is:

- `SessionRuntime` owns foreground session/turn/message projection.
- `TurnLifecycleRuntime` owns foreground turn event-log and session projection
  updates only.
- `RuntimeTaskWorker` plus `RuntimeControlPlane` owns detached, observable,
  cancellable background work.
- Background completions remain task events and are acknowledged through
  `CompletionInbox` / `RuntimeTaskWorker`.

This makes `RuntimeTaskWorker` the only caller that turns detached work into task
ledger rows on the hot path. Ordinary foreground turns remain traceable through
the `turns`, `messages`, event-log, and runtime-event projections, not by being
promoted to task board records.

## Delete Or Merge

- Delete `TurnLifecycle.task_id`.
- Delete `TurnLifecycleRuntime._submit_turn_task`.
- Delete `TurnLifecycleRuntime.complete()` calls to
  `control_plane.succeed(...)`.
- Delete `TurnLifecycleRuntime.interrupt()` calls to
  `control_plane.cancel(...)` / `control_plane.fail(...)`.
- Remove `"agent.turn"` from `ActionKind`.
- Update stale comments/docs that describe `TurnEngine` as an `agent.turn` task.
- Update tests that expect foreground turn ids to be readable as task ids.

## Behavior To Keep

- `turn.started`, `message.inbound`, `turn.completed`, `turn.failed`, and
  `turn.cancelled` event-log entries still exist.
- `SessionRuntime.start_turn()` still records the foreground turn as `running`
  with no task id.
- `SessionRuntime.complete_turn()` still marks foreground turns completed,
  failed, or cancelled.
- Output delivery still stores outbox rows using the turn id as the delivery
  parent reference where current delivery code needs that correlation.
- Background task completion claims still get acknowledged when a synthetic
  completion turn consumes them.
- `RuntimeTaskWorker.start_task()` remains the seam for `agent.spawn`,
  `terminal.exec`, and `evolver.run`.
- `schedule.fire` remains a control-plane task because schedule firing is
  detached host work, not a foreground user turn.

## Why This Is A Demiurge Framework Step

This is not a filter cleanup. It removes the old compatibility assumption that
"every execution record is a task" and makes the runtime vocabulary sharper:
foreground turn, tool call, delivery, schedule fire, and background task each
live in their own projection. The task ledger becomes a deeper module because
its interface only represents controllable detached work; deleting it would now
move real background execution complexity back to child-agent, terminal, and
evolver callers.

# 07:26:48 Hermes-informed redesign plan

## Scope

本轮推进 “demiurge 2.0” 设计中的 approval prompt lifecycle 深化。保留 Agent Core、Agent Slot、Package 能力包概念不变；不修改 authored surface，不修改 package schema，不改变 host-owned approval security policy。

## Hermes 对比

- Hermes 的 blocking approval primitive 在 `tools/approval.py` 中拥有 pending queue lifecycle：register、resolve、clear session、wake blocked entry。Gateway platform adapter 只负责展示按钮或处理 `/approve` / `/deny`，再把选择交回 approval primitive。
- 当前 Demiurge 已在上一轮把 `allow` / `session` / `deny` action semantics 收到 `demiurge.runtime.approvals`，但 pending prompt lifecycle 仍散在 adapter：
  - TUI 用 `_approval_counter`、`_pending_approvals: dict[str, Future]`、`reply_approval()` 和 shutdown loop 维护 lifecycle。
  - Telegram 用 `_approval_counter`、`_pending_approvals: dict[str, TelegramPendingApproval]`、`pending_approval_id`、`_resolve_pending_approval()`、timeout 和 stop cancellation 维护 lifecycle。
- 这使得 adapter interface 仍然浅：transport adapter 不只负责 send/edit，还知道 future creation、id allocation、resolve/cancel/discard/count。

## 设计

扩展 `demiurge.runtime.approvals`：

- `PendingApproval`：保存 `approval_id`、`ApprovalRequest`、`Future[ApprovalDecision]` 和 adapter-owned `payload`。
- `ApprovalPromptRuntime`：
  - `open(request, payload=None)`：生成 id、创建 future、登记 pending entry。
  - `wait(entry, shield=False)`：等待 future；caller 显式调用 `discard()` / `resolve()` / `cancel()` 清理，这样 timeout path 仍能拿到 pending entry 做 transport edit。
  - `get(approval_id)` / `pending_ids()` / `count`：供 adapter 查询状态和测试使用。
  - `resolve(approval_id, decision)`：pop pending entry 并完成 future。
  - `cancel(approval_id, reason)`：deny 并 resolve。
  - `cancel_all(reason)`：shutdown / session cleanup 时 deny 所有 pending entries。
  - `discard(approval_id)`：未 resolve 的异常路径清理。

Adapter 接线：

- `TuiInteractionBridge` 用 `ApprovalPromptRuntime(id_prefix="approval_")` 替换 `_approval_counter` 和 `_pending_approvals` dict。`reply_approval()` 调 shared runtime resolve，`shutdown()` 调 `cancel_all()`。
- `TelegramInteractionBridge` 用 `ApprovalPromptRuntime()` 替换 `_approval_counter` 和 `_pending_approvals` dict。`TelegramPendingApproval` 只保留 transport payload：`source`、`reply_to`、`conversation_key`、`message_id`。
- Telegram `request_approval()` 用 runtime entry 的 `approval_id` 和 future；callback/timeout/stop 路径都通过 runtime resolve/cancel 取得 pending entry，再执行 transport edit。
- 测试改成通过 `pending_ids()` / `count` 读取 pending 状态，避免依赖 adapter-local dict。

## 优势

- locality：future lifecycle、id allocation、resolve/cancel/discard/count 的 bug 集中在一个 module。
- leverage：TUI 和 Telegram 同时复用 pending approval runtime。
- depth：adapter interface 缩小为 “open approval / render transport / wait or resolve / edit transport message”。
- testability：新增 `tests/runtime/test_approvals.py` 覆盖 shared pending lifecycle；现有 TUI/Telegram tests 继续覆盖 wiring。

## Implementation steps

1. 扩展 `demiurge/runtime/approvals.py`，新增 `PendingApproval` 和 `ApprovalPromptRuntime`。
2. 扩展 `tests/runtime/test_approvals.py`，覆盖 open/wait/resolve/cancel/cancel_all/discard/count/id prefix。
3. 更新 `demiurge/ui_gateway/bridge.py` 使用 `ApprovalPromptRuntime`。
4. 更新 `demiurge/channels/telegram/bridge.py` 使用 `ApprovalPromptRuntime`，并把 `TelegramPendingApproval` 降为 payload。
5. 更新相关 TUI/Telegram tests，避免检查 removed adapter-local dict implementation。

## Non-goals

- 不改变 `ApprovalRuntime` 的 security policy、session allowlist 或 event emission。
- 不实现 Telegram `/approve` / `/deny` text fallback。
- 不实现 full Hermes per-session approval queue semantics 或 resolve_all。
- 不改变 Agent Core / Agent Slot / Package 能力包概念或文件格式。

# 12-45-54 Child Agent Invocation Request Plan

## Global architecture review

The previous loop collapsed Agent Slot execution into `SlotPipelineRuntime`.
The next framework-shape problem is the child-agent invocation seam:

- `ModuleAgentsClient` accepts the authored `ctx.agents.run/spawn(...)` calls.
- `RunnerSlotContextHost` forwards the same long argument list to runner private
  methods.
- `SessionTurnStepRunner._run_child_agent(...)` and `_spawn_child_agent(...)`
  forward the same argument list to `ChildAgentRuntime`.
- `ChildAgentRuntime` owns the real lifecycle, but its interface is still a
  wide parameter bundle copied through multiple callers.

This is a shallow module/interface problem. The host should pass a single
runtime-owned invocation object across the seam, not require every caller to
know the exact argument list for child session id, parent turn, slot path,
context fragments, slot selection, bootstrap policy, tool selection, task
notification policy, and pre-resolved child tools.

## Hermes vs Demiurge comparison

- Hermes treats background delegation as a captured task-source block: the
  original goal, context, toolsets, model/session routing, dispatch metadata,
  status, and result summary travel together through the async lifecycle.
- Demiurge already has a stronger Agent Core / Agent Slot / Package concept,
  so it should not copy Hermes' thread executor or global registry. Demiurge's
  host-owned task worker remains the correct execution rail.
- The lesson to adopt is the value-object seam: child-agent lifecycle metadata
  should be carried as one coherent request, then resolved by the runtime.

## Breaking redesign target

Introduce child-agent request objects:

- `ChildAgentRunRequest` captures synchronous child-agent invocation.
- `ChildAgentSpawnRequest` captures background child-agent invocation.
- `ModuleAgentsClient`, `RunnerSlotContextHost`, `ChildAgentRuntime`, and
  `delegate_task` route through those request objects.
- Delete `SessionTurnStepRunner._run_child_agent(...)`.
- Delete `SessionTurnStepRunner._spawn_child_agent(...)`.

Agent Core, Agent Slot, and Package concepts stay unchanged. The authored API
still exposes `ctx.agents.run(...)`, `ctx.agents.spawn(...)`, and
`delegate_task`; only the host-internal child lifecycle interface changes.

## Deep module check

- Interface size: child-agent runtime callers pass one request object instead
  of mirroring every lifecycle argument.
- Deletion test: deleting the request object would scatter child lifecycle
  metadata back across slot context, runner wrappers, delegation tools, and
  tests. The object earns its place by concentrating the host contract.
- Hidden behavior: `ChildAgentRuntime` still owns slot/tool resolution, child
  session creation, metadata shaping, task worker registration, and result
  shaping. The request object is not a pass-through helper; it is the stable
  seam for that lifecycle.
- Adapter discipline: `RunnerSlotContextHost` adapts authored slot ctx calls to
  child-agent runtime requests. Runner remains dependency wiring, not a child
  lifecycle facade.

## Old interfaces and files to delete

- Delete `SessionTurnStepRunner._run_child_agent(...)`.
- Delete `SessionTurnStepRunner._spawn_child_agent(...)`.
- Remove `ChildSlotRequest`, `ChildToolRequest`, `CHILD_AGENT_ALL_TOOLS`, and
  `ResolvedChildAgentTools` imports from runner if no longer needed there.
- Replace direct test usage of runner private child methods with
  `ChildAgentRuntime.run_child(ChildAgentRunRequest(...))`.

## New seam placement

- `demiurge/runtime/child_agents.py`
  - owns `ChildAgentRunRequest` and `ChildAgentSpawnRequest`;
  - exposes `run_child(request)` and `spawn_child(request)`;
  - keeps slot/tool selection and task worker behavior behind this runtime.
- `demiurge/runtime/slot_context.py`
  - `ModuleAgentsClient` constructs request objects from authored
    `ctx.agents.*` calls;
  - `RunnerSlotContextHost` forwards request objects directly to
    `runner.child_agents`.
- `demiurge/runtime/runner.py`
  - wires `ChildAgentRuntime` only;
  - no longer exposes private child-agent lifecycle wrapper methods.

## Why this is framework redesign, not ordinary cleanup

This removes a runner-owned facade around nested agent lifecycle and makes
child-agent invocation a first-class runtime contract. It is a concrete step
toward the new Demiurge shape where runner composes runtimes, authored slots
create runtime requests, and host lifecycle modules own execution semantics.

## Implementation steps

1. Add `ChildAgentRunRequest` and `ChildAgentSpawnRequest`.
2. Change `ChildAgentRuntime.run_child(...)` and `spawn_child(...)` to accept
   request objects.
3. Update `handle_delegate_task(...)` to build a spawn request.
4. Update `ModuleAgentsClient` and `RunnerSlotContextHost` to use request
   objects directly.
5. Delete runner child-agent wrapper methods and stale imports.
6. Update tests that reached into runner private child wrappers.
